# Super Parent V2 Capabilities

This document captures Super Parent capabilities that are intentionally deferred from the first sequential version. The purpose is to preserve the desired behavior without making the first usable slice depend on parallel orchestration.

## Parallel Children

Parallel children are useful when multiple slices are independent and can be verified without stepping on each other. They require additional constraints beyond sequential child orchestration.

### When Parallel Children Are Allowed

Parallel child runs may be created when all of these are true:

- Each child has a disjoint or clearly compatible write scope.
- Each child has independent requirements and verification.
- Child verification does not depend on another child's unmerged changes.
- Merge order is known or can be mechanically resolved.
- The parent can aggregate results and run final integration validation after fan-in.

The default maximum number of parallel children is 4. If more child slices are ready than the concurrency limit allows, the parent should queue the remaining children and start them as running children finish.

Examples that may be parallelizable:

- Adding independent tests in separate frontend and backend test files.
- Investigating unrelated bug reproductions in read-only or mostly read-only children.
- Implementing independent docs or lint checks with non-overlapping files.

Examples that should not be parallelized without stronger coordination:

- Multiple children editing the same shared state model.
- A backend schema change and a frontend type consumer that depends on the schema change.
- Work that requires one child's generated artifact as another child's input.
- Broad refactors across shared modules.

## Required Parallel Components

Parallel execution requires these additional components:

- A dependency graph for slices.
- A declared read/write set for each child.
- Conflict prediction before launch.
- Concurrency limits by cost, runner type, and repository risk.
- A fan-in evaluator that waits for all required children or stops early on critical failure.
- Merge ordering and conflict resolution policy.
- Final integrated validation after accepted children are merged.
- Cancellation or pause policy for sibling children when one child invalidates the shared assumption.

Overlapping file edits are not a hard block. The parent should design slices to avoid merge conflicts, but it must also be able to detect, surface, and resolve conflicts when they arise.

Parallel children still merge only into the parent branch. If two accepted children conflict, the parent resolves the conflict in its own worktree before continuing fan-in or launching dependent children.

The parent may continue unrelated child execution while one child is waiting for human input. The parent owns the dependency graph and must not continue dependent work until the waiting child is resolved.

## Fan-In Outcomes

After parallel children finish, the parent must classify the batch:

- `all_passed_integrate`: all children passed and can be integrated.
- `partial_pass_replan`: some children passed, but failures change the next step.
- `blocked`: one or more children could not verify due to environment or missing information.
- `conflict`: child changes cannot be merged automatically.
- `invalidated`: one child disproved an assumption that made other child work irrelevant.

The parent must not launch the next parallel batch until it has processed fan-in and updated the target inventory.

## Parallel Verification

Each child can verify locally in parallel, but the parent must run integrated validation after fan-in. This catches failures caused by independently correct changes interacting badly.

For high-risk work, the parent should also require a pre-merge review of each child result before integration.

## State Model Additions

Parallel execution extends the v1 parent/child state model by allowing multiple active children at once. That requires the parent reducer to reason about batch state, not just the next child state.

Additional legal steady-state combinations:

- `active` parent with multiple active children whose write scopes and dependencies are compatible.
- `active` parent with a mix of active, paused, completed, and failed children while a fan-in batch is still being evaluated.
- `paused` parent with all active children paused and the batch dependency graph preserved for resume.

Additional illegal steady-state combinations:

- A new parallel batch launched before the prior batch has been evaluated and integrated or rejected.
- Dependent child work launched before the prerequisite child result is accepted and merged.
- Parent completion while any parallel batch has unresolved fan-in state.

## V2 Success Criteria

Parallel Super Parent should not be considered complete until it can demonstrate:

- Independent children can run concurrently without losing child state or evidence.
- The parent can pause all active children and resume the batch without losing dependency state.
- Accepted child branches merge into the parent branch in a deterministic order.
- Conflicting accepted children are resolved and then revalidated on the integrated parent branch.
- A failed or invalidating child causes the parent to stop, replan, or cancel sibling work according to the dependency graph.
- Final validation runs on the integrated parent branch after fan-in.
