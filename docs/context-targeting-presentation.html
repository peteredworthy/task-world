Recommended Action Plan

Updated based on 3-run analysis — the real cost drivers are cache sharing loss, retry tax, and output growth

Immediate Fix — Highest ROI
1

Fix the persistent fan-out failure (step-05 child)

One fan-out child failed 2× in both V1 and V2, burning ~2.3M extra tokens per run in retries. The baseline never failed here. Identify the root cause (likely ambiguous prompt or missing context) and harden specifically for this step. Zero infrastructure changes needed.

−12%
Architecture Fix — Restore Cache Sharing
2

Shared-prefix + step-specific suffix for fan-out

The baseline's real advantage: all fan-out children share the same large context prefix, so it's written to cache once and read N times cheaply. Scoped V2 broke this — each child got a unique prefix, creating N separate cache writes. Fix: give all children an identical base context (full shared docs), append a tiny per-step annotation. Preserve the 82% cache amortization efficiency.

−22%
3

Output verbosity constraints

Step files produced by V2 are measurably larger than baseline, causing the Final Check (S-06) to process 5.2M tokens vs 2.5M. Add explicit token budgets or conciseness instructions to step-writing prompts ("max 400 words per task"). Output token growth compounds into cache creation for every downstream stage.

−11%
Medium Investment
4

Model tiering for dry-run fan-out (S-05)

The dry-run simulation stage (5.8M tokens in V2) is primarily read-and-summarize work. Switching these fan-out children from Sonnet to Haiku reduces per-token cost 5-8×. Cache creation tokens don't change but the cost per token drops dramatically. No prompt changes required.

−9%
5

Fix annotation quality — the extraction script never fired

Root cause of V2's excess: the ctx/ packets are 11,818 bytes vs architecture.md at 12,046 bytes (98% retained). Every H2 section was tagged steps: S-01…S-06 — all steps — so the extractor had nothing to exclude. Scoping never actually happened. Fix: (a) add a post-extraction check that fails if any packet exceeds 70% of the source file size; (b) annotate at H3/H4 granularity (individual fixes/components), not H2; (c) prompt the annotation LLM with explicit examples of narrow tags like steps: S-03.
