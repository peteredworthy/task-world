Super Parent Routine Mechanics

Implementation review and capability report for the reusable parent routine model.

Review Outcome

The implementation now has the deterministic mechanics needed for a sequential Super Parent to create child runs, collect validated evidence, merge accepted work into the parent branch, and block terminal success until inventory and integrated validation are recorded.

Slide 3

Evidence Contract

Acceptance now depends on a full run.evidence.v1 bundle, not only an outcome string.

Accepted Evidence

  • Valid schema version and required fields.
  • Structured command and test evidence.
  • Explicit target reproduction status.
  • Distinct outcome such as verified_fix or behavior_already_correct.

Rejected Evidence

  • Malformed bundles become invalid evidence paths.
  • Missing evidence blocks acceptance.
  • Revision outcomes never enter the merge queue.
  • bug_not_reproduced requires parent or human decision.

Slide 8

Git Safety

The wrapper is a defense-in-depth layer around run-worktree command use.

Allowed

status, diff, add, restore, commit, log, show, rev-parse, ls-files.

Blocked

Branch switching, merges, rebases, resets, pushes, fetches, ref updates, unsafe flags, git env overrides, and path arguments outside the run worktree.

Slide 9

Ability Expansion

These mechanics make parent-style routines generally authorable, not hard-coded to one controller.

Routine Authors

Can express orchestration policy while relying on service-owned state checks, merges, and evidence validation.

Agents

Can use MCP tools to launch children, update oversight state, and integrate accepted work from clean contexts.

Users

Get a single parent run that reports child progress, human attention needs, final validation, and remaining risk.

Slide 10

Residual Risk

The implementation is now suitable for sequential Super Parent use, with these operational boundaries.

V1 Boundary

Sequential orchestration is supported. Parallel child fan-in, predictive conflict planning, and batch invalidation are still V2 concerns.

Wait Model

Wait intent and observations are written into the parent oversight payload. Final validation markers are stamped by the service and invalidated when the parent HEAD no longer matches the verified commit.
