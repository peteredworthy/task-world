Impact on test execution speed

Coverage instrumentation adds overhead. Here's the real cost breakdown.

Backend — without coverage
~35s
3,019 tests, 8 workers parallel
Backend — with coverage
~65s
Same tests + branch instrumentation
Overhead added
+30s
~86% slower per full CI run
  • ⚡Why so much overhead? Branch coverage (--cov-branch) instruments every conditional, not just line execution. Each branch decision is tracked. With 18,000+ lines across 150+ modules, this is significant syscall overhead even with parallel execution.
  • 💡Mitigation available: Run coverage only on the files changed in a PR — use --cov-source=$(git diff --name-only origin/main | grep .py) in CI. Local development can run pytest -n auto (no coverage) for fast feedback, with full coverage reserved for CI.
  • 🎯Coverage is a merge gate, not a dev loop tool. Developers should run tests without coverage locally. CI enforces the threshold on every PR. The 30s overhead is a CI cost, not a local feedback cost.
  • 📊Frontend overhead is negligible — v8 coverage is native to Node.js; vitest with coverage adds <2s to a 25s test suite.
Slide 5 / 6

Classes of issue now protected

1
Untested transitionsBug 6NEW
Before: a new transition function (transition_to_recovering, transition_force_accept) could ship with zero test coverage. No CI signal. After: any function that drops coverage below 80% fails CI immediately. A new uncovered transition = red build.
2
Silent regression in tested codeNEW
Branch coverage catches conditional paths tests don't exercise. A refactor that adds a new if/else branch to a well-tested function is visible immediately via the branch coverage delta — even if line coverage doesn't change.
3
Coverage erosion over timeNEW
Without thresholds, coverage can drift silently downward as new code is added without tests. The threshold locks in the current floor. Every PR must maintain or improve coverage to merge. Gradual erosion is structurally prevented.
4
Coverage blind spots made visibleSURFACED
Running coverage for the first time exposes that hooks/useWebSocket.ts is at 16% (home of Bug 1), workflow/service.py is at 65%, and the OpenHands Docker agent is at 43%. These were unknown unknowns — now they are documented gaps with line numbers.
5
Not protected: async concurrency racesStill exposed
Coverage thresholds tell you a line executed, not whether it executed safely under concurrent load. Bugs 5 and 8 (WS batch timer races) can have 100% coverage and still be broken. The AST-based linter (Gap A from the strategy doc) is the correct protection class for these.
Key insight

Coverage thresholds are a forcing function, not a correctness guarantee. They don't prove code works — they prove code was executed during tests. Their real value is making untested code structurally impossible to ship without a deliberate decision to lower the bar. That decision is now visible in the diff, not buried in a bug report six months later.

Next steps to reach the thresholds

Current backend coverage is 71.7% — 8.3 points below the 80% target. Here's the highest-leverage path to close the gap.

Module Current Gap Priority Why
workflow/service.py 65% +15% needed P1 All run lifecycle orchestration lives here. Largest untested surface.
runners/signals/runtime.py 62% +18% needed P1 Signal handling drives state recovery; uncovered paths = silent failures.
execution/phase_handler.py 62% +18% needed P2 Build/verify phase transitions. Interacts with transitions.py.
runners/executor.py 69% +11% needed P2 Agent execution lifecycle. Low coverage = runtime surprises.
openhands/docker_agent.py 43% +37% needed P3 External agent — harder to test. Can be excluded from threshold or mocked.
Short-term option
Lower threshold to current baseline

Set --cov-fail-under=71 (current level). Locks in the floor immediately. Raise incrementally as coverage improves. CI green on day one. Allows shipping without blocking all PRs while coverage is improved.

Medium-term option
Pair Priority 2 with Priority 3

Priority 3 (WS processEvent tests) directly covers useWebSocket.ts — the most critical untested hook. Adding those tests would bring the frontend from 38% toward 70% on the hooks layer. Backend needs focused integration tests for service.py paths.

  • 📌The 80% backend threshold is the right long-term target. Short-term, start at 71% (current) and raise by 2% per sprint as tests are added. The threshold change itself is a one-line edit.
  • 📌The 70%/65% frontend thresholds require ~40 points of new coverage. The hooks layer is the gap. Priority 3 (WS unit tests) + fixing useApi test coverage are the highest leverage investments.
  • 📌Coverage data is now baseline-captured. The term-missing report shows exact line numbers for every uncovered path. Future contributors have a map, not just a number.
