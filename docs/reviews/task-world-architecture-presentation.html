Task-World architecture review

Current architecture: disciplined intent, hybrid reality

Task-World is a routine-driven orchestration system for LLM coding agents. The intended design is modular, event-backed, worktree-isolated, and signal-driven; the operational risk comes from the layers required to keep that design compatible with existing runs, external agents, projections, and parent/child workflows.

Executive view

The system has a clear conceptual core, but correctness depends on several duplicated representations staying aligned.

Routine snapshot

YAML routines are validated into Pydantic models, assembled from optional external step files, and embedded into each run as routine_embedded.

Run worktree

Execution occurs on per-run branches and worktrees. Attempts carry commit metadata so builder and verifier phases can reason about changes.

Events plus projections

The event stream is the durable write path for current workflow changes, while ORM tables are materialized read models with some compatibility paths.

Signal queue

Lifecycle controls are queued as events and consumed by a per-run signal consumer, giving the system recovery semantics after crashes.

Most important caveat: Task-World is event-backed, not purely event-sourced. Compatibility mutations, embedded snapshots, direct apply methods, and legacy replay expansion are intentional but increase drift risk. Some architecture docs also lag the code, especially around runner variants.

The system is organized around a small number of top-level modules, with workflows coordinating config, state, database projections, git worktrees, and runner backends.

How routines work

A routine is the task template. A run is a materialized instance of that template with generated runtime IDs, selected runner config, worktree metadata, and progress state.

Key constraints: existing runs keep the embedded snapshot; ref and use are rejected; external step files are supported but exclusive with inline step fields; git versioning records a dirty flag rather than forcing clean commits. Evidence: config models, loader, discovery, factory, and run API paths in src/orchestrator/config, src/orchestrator/state/factory.py, and src/orchestrator/api/routers/runs.py.

Builder/verifier execution loop

Each task moves through a builder phase and verifier phase with fresh LLM context. The orchestrator gates the transition, retries revisions, and records attempts.

Normal phase progress Accepted result Revision or human path Failure path

Run lifecycle state machine

The lifecycle adds stopping as an observable intermediate state so a pause can be requested while an agent process is being cancelled.

Lifecycle invariants

  • start_run enqueues RUN_START; the consumer applies draft -> active.
  • pause_run first applies active -> stopping, then enqueues PAUSE.
  • resume_run enqueues RESUME; the consumer applies paused -> active.
  • cancel_run enqueues CANCEL; the consumer applies terminal failure.
  • Executor failures, stale-run startup repair, and clarification response handling can call apply_* methods directly.

Evidence: WorkflowService, SignalConsumer, RunWorkflow, api/app.py, and api/routers/clarifications.py expose this split between enqueue methods and apply methods.

Event and projection flow

Workflow changes are appended to events_v2, synchronously projected into read-model tables, written to the JSONL history outbox, and broadcast to connected UIs.

Strength

Projection failures abort append transactions, so the event log and read models are intended to stay consistent.

Migration scar

Legacy run snapshots can be expanded into child projection events during replay.

Risk

Every new event type needs matching event schema, command handler, projector behavior, and tests.

Signal queue model

Signals are one-time control commands persisted as events. They let API calls request lifecycle changes without directly owning the executor loop.

Signal types

SignalPurpose
RUN_STARTRegister and start a run workflow.
RESUMERestart a paused workflow, optionally with a new runner config.
PAUSESet paused state and stop executor progress.
CANCELMark run failed and clean up execution.
ACTIVITY_COMPLETEDExternal activity completed builder phase.
ACTIVITY_VERIFIEDExternal activity completed verifier phase.

Delivery semantics

  • SignalEnqueued is appended first.
  • The consumer processes pending signals FIFO per run.
  • SignalProcessed is appended only after handler success.
  • If a handler fails, the signal remains eligible for redelivery on restart.
  • A signal can interrupt between runner phases, not always mid-subprocess.

This model is strong for crash recovery but creates timing cases: an API call can return before the projected lifecycle transition has been consumed.

The central weakness: state has many shapes

The architecture trades isolation and recovery for duplication. The highest-risk bugs appear where these shapes diverge.

Architectural consequence: Tests that prove one layer behaves correctly may not prove that event replay, API presentation, UI state, and worktree side effects stay aligned. This is the dominant source of accidental breakage.

Why effective tests are hard

The project correctly forbids broad mocking, which improves confidence but forces many real integration surfaces into tests.

Async control flow

  • Signals can be queued before they are applied.
  • Consumer and executor loops run concurrently per run.
  • Agent output is persisted out of band.

Durable replay

  • Tests need to cover event append and projection rebuild.
  • Old events may expand into modern projection events.
  • Read-model corruption tests are valuable but not exhaustive.

External execution

  • Worktrees require real git repos and branch operations.
  • Agent runners depend on subprocesses, SDKs, local servers, and callbacks.
  • UI sees async states through REST, SSE, and WebSocket surfaces.

Testing implication

The most protective tests are not small unit tests of transition helpers alone. They are focused integration tests that append events, rebuild projections, drive lifecycle signals, touch real worktrees, and assert the API shape the UI consumes.

What can and cannot be changed

The architecture is changeable, but changes have different blast radiuses depending on whether they touch templates, runtime state, durable events, or worktree behavior.

Low to moderate blast radius

  • Routine text, requirements, and rubrics for future runs.
  • New task-level auto-verify commands that obey validation rules.
  • UI presentation changes that do not alter REST schemas.
  • New public exports from module roots, with import checks passing.

High blast radius

  • Event payload shape or event type names.
  • Task/run state enum changes.
  • Run lifecycle semantics around stopping, pause, resume, or cancel.
  • Runner callback contracts and external REST/MCP callback behavior.
  • Worktree branch naming, reset, commit, or merge assumptions.
Rule of thumb: if a change crosses from config into runtime state, event payloads, DB projection logic, or worktree operations, design the migration and replay path first. Otherwise tests can pass on new runs and still break existing ones.

Recommended architecture safeguards

These are scoped to the current architecture; they aim to reduce accidental breakage without a rewrite.

Strengthen contracts

  • Maintain an explicit event compatibility matrix: event type, projector, API surface, replay test.
  • Promote signal-routing checks from placeholder to active AST rules for direct lifecycle mutations outside allowed files.
  • Keep enum exports generated and compare backend/frontend state names in CI.
  • Document which apply_* methods are consumer-only versus executor self-pause paths.

Improve protection

  • Add golden replay scenarios for old and current runs after each event schema change.
  • Exercise pause/resume/cancel via API plus signal consumer, not only service methods.
  • Keep runner contract tests backend-neutral, then add small backend-specific failure cases.
  • Test routines with external step files and embedded routine snapshots across schema changes.

Bottom line

Task-World's architecture is coherent but fragile at boundaries. The best next improvements are boundary checks, replay tests, and narrower runner contracts rather than broader abstractions.
