What The UI Currently Conveys

Run-list modelParent and child runs use the same standard card. Status, routine, branch, progress, cost, and task metadata are visible.
Parent oversightThe detail page shows child count, next action, terminal blocked/clear, child counts, blockers, attention, and child summaries.
Child contextChild detail shows a compact parent link and slice ID, then reuses normal run review and diff surfaces.
  • The UI confirms hierarchy exists, but mainly after opening detail pages.
  • The parent panel is useful as a compact status widget.
  • The child page remains optimized for normal run review instead of parent mission review.

Super Parent UI Review

Primary Jobs To Be Done

StartGive the system a large goal, artifacts, and child-run budget.
UnderstandSee what the parent believes about scope, risks, ready slices, and blockers.
MonitorTrack child runs in the context of the parent mission.
DecideAccept, reject, retry, or abandon child evidence.
ValidateKnow why completion is blocked and review final proof.
  • The current flow supports these jobs mechanically, but several jobs require users to infer state from normal run surfaces.
  • The strongest opportunity is turning parent orchestration into an explicit workspace instead of a normal run with an added panel.

Super Parent UI Review

Main Gaps

1. Hierarchy is hiddenParent and child runs are flat on the dashboard. The relationship is visible only after opening details.
2. Mission state is hiddenCurrent understanding, target inventory, decisions, slices, attempts, and final validation are not rendered.
3. Attention is incompleteNeeds Input ignores oversight attention, stalled slices, illegal states, and terminal guard blockers.
4. Acceptance is evidence-lightThe accept path does not consolidate test evidence, files changed, uncertainties, inventory impact, and merge risk.

Workflow Opportunity

IntakeMission, artifacts, budget, evidence policy.
PlanInventory and candidate slices.
DispatchLaunch child with expected evidence.
MonitorGrouped child progress and attention.
DecideEvidence review and accept/reject/retry.
ValidateTerminal guard and final report.

The goal is to make the parent workflow visible as a product flow, while preserving normal run review where it already works.

Iteration Focus

  • Add parent/child grouping and oversight attention to the dashboard.
  • Create a parent workspace with Understanding, Inventory, Children, Decisions, Evidence, and Validation sections.
  • Add parent context to child detail: slice goal, affected inventory, expected evidence, and acceptance consequences.
  • Replace evidence-light accept actions with a decision modal explaining evidence, merge impact, and unresolved questions.
  • Expose oversight freshness and final validation/report state.
Open questionsWho is the primary operator? Should users approve each slice launch? What evidence standard should block acceptance? Can unresolved inventory be waived?
