task-world · execution-graph evaluation · 2026-06-14

How effective, efficient & expensive is the graph approach?

A controlled, big-task comparison of three orchestration strategies — graph (adaptive horizon planning), idea-to-plan (front-loaded), and mind-the-gap (one chunk at a time) — building the same non-trivial feature with the same worker model, graded by the same hidden 56-test oracle. Follow-on to carrier-comparison.md, which compared carriers on a one-function task and could not separate them.

The result in one chart

Correctness (hidden acceptance tests passed, out of 56) against full cost (USD, codex tokens). Bubble area ∝ number of agent invocations. Up-and-left is better.

Only adaptive horizon planning (graph) reached 100%. It closed the last 5 cross-feature interaction bugs a single shot missed — at ~4.5× the one-shot cost. Spending more than that (mind-the-gap, finest granularity) scored below one-shot: more cycles ≠ more correct.

What was built (the target feature)

Versioned File Workspace

Extension of a small FastAPI file-manager (desktop-test). Deliberately beyond one-shot: eight capabilities whose interactions are the hard part.

  • Version history — snapshot on overwrite, list/read/restore/diff, 20-version retention
  • Trash bin — soft delete, restore w/ collision-rename, purge, dirs
  • Search — name/content/tag/type/size/prefix filters
  • Move & copy — move carries history+tags; copy duplicates content+tags, not history
  • Tagging — follows rename/move; copy duplicates; purge drops
  • Quota — live-bytes cap → 413 with no partial write; versions/trash excluded
  • Activity log — bounded ring of mutations
  • Interactions — trash carries history+tags; restore respects quota; rename carries history…

Calibration — making it hard enough

A naive one-shot is the difficulty gate. Hardened until a single agent could no longer ace it.

The experiment: one axis, three strategies

The strategies differ on a single dimension — plan granularity — the very dimension the graph approach was designed around (adaptive plan granularity). Held constant: target feature, worker model gpt-5.3-codex-spark, orchestrator (Claude Opus), fresh code copy per arm, and the 56-test grading oracle.

One-shot
1 agent, no plan. The baseline.
Idea-to-plan
Plan everything up front → build all → verify once.
Graph (horizon)
Plan a multi-step region → build+verify → replan next region. Adaptive batches.
Mind-the-gap
Plan one small chunk → build → validate → repeat. Finest grain.

Arrow of increasing planning granularity → more, smaller planning turns. A fifth run (A′) exercised task-world's real graph carrier on the same feature as a grounding datapoint.

What each strategy actually did

A — Graph (horizon) · 56/56 · best

17 agents across 5 planned regions, an independent verifier node per region. The horizon planner staged all eight capabilities; the verifier caught and bounced incomplete regions.

Honesty note: first graded 45/56 because a verdict-parse bug in our driver aborted the loop after the final region was planned but before it was built. Running that planned region (verifier passed first try) closed to 56/56. The 45 is a harness artifact; strategy correctness is 56/56.

B — Idea-to-plan · 52/56

3 agents: plan → implement → verify. Cheapest of the verified arms ($2.23). Its single up-front plan + one verify pass left 4 trash/directory interaction bugs — its own tests passed, so the front-loaded verifier never saw them.

Front-loading bought almost nothing over one-shot here (+1 test), at +60% cost.

C — Mind-the-gap · 48/56 · most cost

29 agents over 9 chunk cycles (fresh builder+validator each). The most expensive arm — yet scored below one-shot.

Finest granularity backfired: narrow per-chunk validators rubber-stamped chunks carrying latent interaction bugs (activity-log shape, quota no-partial, copy-history). Verification scope matters more than cycle count.

A′ — Real graph carrier · 33/56

task-world's actual graph carrier (native codex worker) completed the run — 11 graph nodes, verifier passed. But a single-task routine compiles to a minimal graph (one worker + one verifier, no planner nodes), so it behaved like a one-shot worker on the routine's condensed context and the verifier passed coarse requirements, missing quota/activity/interactions.

Two caveats: it needed manual resume through transient "graph quiescent" stalls, and it received thinner input (routine task_context) than the other arms (full spec) — so 33/56 partly reflects input asymmetry, not just the carrier.

Findings

1 · Cost tracks planning granularity, not correctness

$1.4 → $2.2 → $6.3 → $6.4; agents 1 → 3 → 17 → 29. Finer planning = more agents = more turns = more cost, monotonically. Correctness does not follow the same curve.

2 · Adaptive horizon planning is the only arm at 100%

The graph strategy bought the last five cross-feature interaction bugs a one-shot leaves — the exact "large/ambiguous task" payoff the earlier study predicted but couldn't show on a toy.

3 · More iteration is not more correctness

Mind-the-gap spent the most and scored the lowest of the loops. Independent verification only helps if each verifier's scope spans the interactions; per-chunk validators are too narrow.

4 · The carrier's value needs real planner + verifier nodes

On a thin single-task routine the real graph degenerates to one-shot-plus-a-coarse-verifier (33/56) and adds operational overhead. The graph approach's benefit (Arm A, 56/56) comes from horizon planner nodes and detailed staged verification — which that routine never compiled to.

Decision-relevant takeaway. The graph approach earns its premium specifically on maximum correctness for interaction-heavy work, and only when compiled with planner nodes + adequately-scoped verifiers. For cheap "good enough" output, idea-to-plan or even one-shot dominate on $/test. The lever that decides the outcome is verification scope, not iteration count.

Caveats — how far to trust this

  • n = 1 per arm. Codex is non-deterministic; treat deltas as directional, not significant — same posture as the prior small-task study.
  • Level mismatch. Arms A/B/C are orchestrator-driven strategies on a common harness; A′ alone is task-world's productionized carrier. The clean controlled comparison is A vs B vs C; A′ grounds the strategy in the real system.
  • Input asymmetry for A′. A′ got the routine's condensed task_context; the others got the full spec — depressing A′'s score independent of the carrier.
  • Token capture. codex_server (A′) does not persist token usage — a known measurement gap; A′ cost is by node/agent proxy only.
  • Harness artifact, corrected. Arm A's first 45/56 was our verdict-parser aborting before the planner's final region built; reported transparently and corrected to 56/56.
