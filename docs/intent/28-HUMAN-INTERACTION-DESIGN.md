# Human Interaction Design

**Created:** 2026-02-06
**Status:** Approved for implementation

This document defines the design for human interaction patterns in the orchestrator workflow.

---

## Overview

There are two distinct human interaction patterns:

1. **Question Answering (Clarification)** - Builder needs human input to proceed
2. **Confirmation Before Action (Approval Gate)** - Human reviews and approves before proceeding

Both patterns use a new task status: `PENDING_USER_ACTION`

---

## Pattern 1: Question Answering (Clarification)

### Purpose
The Builder (LLM) encounters ambiguity or needs decisions during execution. It generates questions for the human to answer. Once answered, the Q&A becomes an artifact that can be referenced in subsequent build and verification phases.

### Flow
```
Builder executing
    ↓
Builder generates questions (dynamically, not from YAML)
    ↓
Task status → PENDING_USER_ACTION
    ↓
UI/CLI presents questions to human
    ↓
Human answers (multi-choice + free text option per question)
    ↓
Answers appended to clarifications artifact file
    ↓
Task status → BUILDING (continues with artifact available)
```

### Artifact Storage
- **Location:** File in worktree, e.g., `docs/{feature}/clarifications.md`
- **Format:** Markdown with structured Q&A sections
- **Accumulation:** Q&A appended across attempts, forming a growing artifact
- **Prompt Integration:** Builder/verifier prompts should advertise this file location

### Question Format
Each question includes:
- `id`: Unique identifier
- `question`: The question text
- `context`: Why this clarification is needed
- `options`: List of multi-choice options (2-4 options)
- `allows_free_text`: Boolean (always true - user can provide custom answer)

### Data Model

```python
class ClarificationQuestion(BaseModel):
    """A question from the builder needing human input."""
    id: str
    question: str
    context: str
    options: list[str]  # Multi-choice options
    # User can always provide free text instead

class ClarificationAnswer(BaseModel):
    """Human's answer to a clarification question."""
    question_id: str
    selected_option: str | None  # If chose from options
    free_text: str | None  # If provided custom answer
    answered_by: str
    answered_at: datetime

class ClarificationRequest(BaseModel):
    """A set of questions from builder needing answers."""
    id: str
    run_id: str
    task_id: str
    attempt_num: int
    questions: list[ClarificationQuestion]
    created_at: datetime

class ClarificationResponse(BaseModel):
    """Human's answers to a clarification request."""
    request_id: str
    answers: list[ClarificationAnswer]
    responded_at: datetime
```

### Artifact File Format

```markdown
# Clarifications Log
<!-- Auto-generated by Orchestrator. Referenced in build/verify phases. -->

## Clarification 1 (Step S-01, Attempt 1)
**Requested:** 2026-02-06T10:30:00Z
```

## Pattern 2: Confirmation Before Action (Approval Gate)

### Purpose
After the Builder completes work and the Verifier runs automated checks, the human reviews a summary of what was done and decides whether to accept or reject the verification.

### Flow
```
Builder completes work
    ↓
Builder outputs summary artifact to step-specified location
    ↓
Verifier runs automated checks (auto_verify, grade checks, etc.)
    ↓
If automated checks pass:
    ↓
Task status → PENDING_USER_ACTION
    ↓
UI/CLI presents summary (markdown rendered) to human
    ↓
Human accepts or rejects
    ↓
Accept: Task status → COMPLETED, proceed to next task/step
Reject: Task status → BUILDING, attempt++ (back to builder for revision)
```

### Summary Artifact
- **Location:** Specified in step/task config (e.g., `summary_path: "docs/{feature}/step-summary.md"`)
- **Format:** Markdown
- **Content:** Builder-generated summary of work completed
- **Verifier Access:** Verifier reads this artifact as part of verification

### Configuration

```yaml
steps:
  - id: "S-02"
    title: "Human Review"
    gate:
      type: human_approval
      summary_artifact: "docs/{{feature}}/plan-summary.md"  # Where builder writes summary
      approval_prompt: |  # Optional additional context for reviewer
        Review the generated plan artifacts and confirm alignment with intent.
      require_comment: false  # Whether rejection requires explanation
    tasks:
      - id: "T-01"
        # ...
```

### Scope
- **Per-attempt:** Each attempt is an independent approval request
- **Rejection:** Returns task to BUILDING for another attempt (not failure)
- **History:** Each attempt's approval/rejection is tracked independently

## Task Status Extension

Add to `TaskStatus` enum:

```python
class TaskStatus(str, Enum):
    PENDING = "pending"
    BUILDING = "building"
    PENDING_USER_ACTION = "pending_user_action"  # NEW
    VERIFYING = "verifying"
    COMPLETED = "completed"
    FAILED = "failed"
```

### Status Transitions

```
BUILDING → PENDING_USER_ACTION  (builder needs clarification OR verification passed awaiting approval)
PENDING_USER_ACTION → BUILDING  (clarification answered, continue building OR approval rejected)
PENDING_USER_ACTION → VERIFYING (clarification answered during verify phase)
PENDING_USER_ACTION → COMPLETED (approval accepted)
```

## UI Design

### Dashboard Integration

1. **Filter:** Add "Needs Input" filter option on dashboard run list
2. **Notification Badge:** Visual indicator (badge/icon) when runs need human input
3. **Run Card:** Show indicator on run cards that have pending user actions

### Modal Dialog

Triggered when user clicks on a task in `PENDING_USER_ACTION` status.

#### For Confirmation (Approval)
- Render summary artifact as markdown
- Show approval prompt if configured
- "Approve" and "Reject" buttons
- Optional comment field (required if `require_comment: true`)

#### For Question Answering (Clarification)

**Layout:**
```
┌─────────────────────────────────────────────────────────┐
│ Clarification Required                          [×]     │
├─────────────────────────────────────────────────────────┤
│ ○ One at a time  ● All at once          [toggle]        │
├─────────────────────────────────────────────────────────┤
│                                                         │
│ ▼ Q1: Should auth support OAuth2 or password?           │
│   Context: Requirements mention "secure login"...       │
│   ┌─────────────────────────────────────────────────┐   │
│   │ ○ OAuth2 only                                   │   │
│   │ ○ Username/password only                        │   │
│   │ ● Both OAuth2 and username/password             │   │
│   │ ○ Other: [________________]                     │   │
│   └─────────────────────────────────────────────────┘   │
│                                                         │
│ ▶ Q2: What rate limit is acceptable?                    │
│   (collapsed - click to expand)                         │
│                                                         │
│ ▶ Q3: Which database for sessions?                      │
│   (collapsed - click to expand)                         │
│                                                         │
├─────────────────────────────────────────────────────────┤
│                              [Cancel]  [Submit Answers] │
└─────────────────────────────────────────────────────────┘
```

**Interaction Behavior:**

| Mode | Behavior |
|------|----------|
| One at a time | Only current question expanded, others collapsed |
| All at once | All questions expanded, no auto-collapse |

| Action | Result |
|--------|--------|
| Answer question | Auto-contract current (unless user manually expanded), expand next, scroll to next |
| User manually expands | Keep expanded after answering |
| User manually contracts | Still expand when it becomes "next" question |
| Toggle mode | Reflow all questions to match new mode |

## CLI Design

### Command: `runs approve <run_id>`

Interactive mode that:
1. Lists pending user actions for the run
2. For each pending action:
   - **Confirmation:** Shows rendered summary, prompts Accept/Reject
   - **Clarification:** Steps through questions one by one

### Example Session

```
$ orchestrator runs approve run-abc123

Run run-abc123 has 1 pending action:

Step S-02 "Human Review" - Task T-01 "Await Human Feedback"
Type: Clarification (3 questions)

─────────────────────────────────────────────────────────

Q1/3: Should authentication support OAuth2 or username/password?

the auth method.

─────────────────────────────────────────────────────────
```

## API Extensions

### New Endpoints

```
POST /api/runs/{run_id}/tasks/{task_id}/clarifications
  - Builder submits questions needing answers
  - Body: { questions: [...] }
  - Transitions task to PENDING_USER_ACTION

POST /api/runs/{run_id}/tasks/{task_id}/clarifications/{request_id}/respond
  - Human submits answers
  - Body: { answers: [...] }
  - Writes to artifact file
  - Transitions task back to BUILDING

GET /api/runs/{run_id}/tasks/{task_id}/clarifications/pending
  - Get pending clarification request for a task

POST /api/runs/{run_id}/tasks/{task_id}/approve
  - Human approves verification (for approval gates)
  - Body: { approved: true, comment?: string }
  - Transitions task to COMPLETED

POST /api/runs/{run_id}/tasks/{task_id}/reject
  - Human rejects verification (for approval gates)
  - Body: { reason?: string }
  - Transitions task back to BUILDING, attempt++

GET /api/runs/{run_id}/pending-actions
  - List all pending user actions across the run
  - Returns: tasks in PENDING_USER_ACTION with type (clarification/approval)
```

## Prompt Integration

### Clarifications File Advertisement

Update builder/verifier prompts to include:

```
## Clarifications

Previous clarifications from the human are recorded in:
  {clarifications_path}

Review this file for context on decisions made. If you need additional
clarification, use the request_clarification tool.
```

### MCP Tool for Clarification

Add tool for builder to request clarification:

```python
@mcp_tool
def request_clarification(
    questions: list[dict],  # [{question, context, options}, ...]
) -> str:
    """
    Request clarification from the human.

    The task will pause until the human answers. Answers will be
    appended to the clarifications artifact file.

    Args:
        questions: List of questions, each with:
            - question: The question text
            - context: Why you need this clarification
            - options: 2-4 suggested answers (user can also provide custom)

    Returns:
        Confirmation that clarification was requested
    """
```

## Database Schema Extensions

### New Tables

```sql
-- Clarification requests
CREATE TABLE clarification_requests (
    id TEXT PRIMARY KEY,
    run_id TEXT NOT NULL REFERENCES runs(id),
    task_id TEXT NOT NULL,
    attempt_num INTEGER NOT NULL,
    questions JSON NOT NULL,  -- List of ClarificationQuestion
    created_at TIMESTAMP NOT NULL,
    responded_at TIMESTAMP,
    UNIQUE(run_id, task_id, attempt_num)
);

-- Clarification responses
CREATE TABLE clarification_responses (
    id TEXT PRIMARY KEY,
    request_id TEXT NOT NULL REFERENCES clarification_requests(id),
    answers JSON NOT NULL,  -- List of ClarificationAnswer
    responded_by TEXT NOT NULL,
    responded_at TIMESTAMP NOT NULL
);
```

### Extended Task Model

```python
class TaskState(BaseModel):
    # ... existing fields ...
    pending_action_type: str | None = None  # "clarification" | "approval"
    pending_clarification_id: str | None = None
```

## Configuration

### Global Config Extension

```yaml
# ~/.orchestrator/config.yaml
clarifications:
  default_artifact_path: "docs/clarifications.md"  # Default if not specified
  timeout_minutes: 60  # Auto-fail if no response (optional)
```

### Routine Config

```yaml
routine:
  clarifications:
    artifact_path: "docs/{{feature}}/clarifications.md"

  steps:
    - id: "S-02"
      gate:
        type: human_approval
        summary_artifact: "docs/{{feature}}/review-summary.md"
        require_comment: false
```
