What is Task-World?

Task-World is an AI agent orchestration platform that manages the full lifecycle of software engineering tasks — from planning through implementation to verification. It coordinates multiple AI agents (Claude CLI, Claude SDK, Codex, OpenHands) to build code, verify correctness, and iterate on feedback, all within isolated git worktrees.

The system decomposes work into routines (YAML-defined workflows) containing steps (sequential phases) and tasks (parallelizable units). Each task goes through a builder→verifier cycle with auto-verification, grading rubrics, and retry logic. A real-time dashboard provides visibility into runs, diffs, test results, and agent progress.

Multi-Agent Orchestration Registry-based agent architecture with auto-discovery. Supports Claude CLI, Claude SDK, Codex Server, OpenHands, and user-managed agents.
Builder/Verifier Cycle Tasks go through build→verify→grade cycles with configurable max attempts, auto-verification scripts, and LLM-based grading rubrics.
Git Worktree Isolation Each run gets its own git worktree, providing filesystem isolation between concurrent agent executions with branch management.
YAML-Defined Routines Declarative workflow definitions with step conditions, fan-out expansion, repeat-for loops, and embedded routine support.
Real-Time Dashboard React frontend with WebSocket-driven live updates, diff review, test execution, conflict resolution, and activity timelines.
Auto-Verification File existence checks, JSON validation, script execution, and LLM grading with critical/expected/nice priority levels.
Event-Sourced State All state changes emit immutable events persisted to an event store. Enables audit trails, recovery, and real-time UI sync.
MCP Integration Model Context Protocol server exposes tools to external agents for run status, task context, and clarification handling.

Consolidation Plan: 19 → 9 Modules Aggressive simplification with clear boundaries

The current 19 top-level modules include many “phantom modules” — tiny, single-consumer packages that inflate the dependency graph without providing meaningful boundaries. This plan absorbs 10 modules into their natural parents, producing 9 focused top-level packages with clean, obvious ownership.

Phase 1: Zero-risk absorptions — no external consumers, mechanical moves

M1: Delete routers

orchestrator/routers/ → deleted
6 LOC legacy re-export shim. Nothing imports from it — verified via grep across the entire codebase. Pure dead code.
Zero risk — 0 files to update

M2: Absorb scaffolding into runners

orchestrator/scaffolding/ → orchestrator/runners/scaffolding/
Scaffolding (template copying, .gitignore setup, workspace prep) is used exclusively by runners/executor.py. Zero external dependents. Already conceptually part of the execution pipeline.
Low risk — ~1 import path in executor.py

M3: Absorb cache into git

orchestrator/cache/ → orchestrator/git/cache.py
88 LOC LRU cache used only by git/cached_diff_ops.py and api/routers/review.py (for git diff caching). It’s a git implementation detail. The review router already depends on git — no new dependency edge.
Low risk — 2 import paths to update

M4: Absorb artifacts into workflow

orchestrator/artifacts/ → orchestrator/workflow/artifacts/
146 LOC, self-contained artifact registry. Only consumed by workflow/context_builder.py and runners/executor.py. Both already depend on workflow. Artifacts are produced during the workflow lifecycle — they belong here.
Low risk — ~2 import paths to update

Phase 2: Semantic consolidations — single-consumer modules joining their natural parent

M5: Absorb review into git

orchestrator/review/ → orchestrator/git/review/
261 LOC (test_runner.py + models.py) providing test execution in git worktrees. Only consumer is api/routers/review.py, which already imports from 5 git sub-modules. Consolidation reduces the API router’s external dependency count from 2 (git + review) to 1 (git).
Low risk — ~3 import paths to update

M6: Absorb repos into git

orchestrator/repos/ → orchestrator/git/repos/
268 LOC for git repository discovery and branch introspection. Self-contained (no dependencies beyond itself). Only consumed by api/routers/repos.py, mcp/tools.py, and cli. Semantically inseparable from git operations — it answers “what git repos exist and what branches do they have?”
Low risk — ~3 import paths to update

M7: Absorb metrics into api

orchestrator/metrics/ → orchestrator/api/metrics.py
70 LOC cost estimation from token data. Only used by api/routers/runs.py to populate cost fields in responses. It’s a presentation concern — transforming raw token counts into dollar amounts for the API consumer.
Low risk — 1 import path to update

M8: Absorb mcp into api

orchestrator/mcp/ → orchestrator/api/mcp/
785 LOC MCP server mounted by api/app.py. It’s an alternative API transport (like WebSocket) — it exposes workflow tools to external agents over SSE. Currently depends on workflow and repos (which will be in git). Keeping all external interfaces in api creates a single entry-point layer.
Low risk — 1 import in app.py to update

M9: Absorb agents into runners

orchestrator/agents/ → orchestrator/runners/profiles/
348 LOC agent configuration CRUD (profiles, seeding Planner/Builder/Verifier defaults). Only consumed by api/routers/agents.py and app.py (for seeding). Runners already contains runners/agents/ for agent implementations — the config/profiles layer belongs alongside them. Renaming to profiles avoids collision with the existing runners/agents/ sub-package.
Low risk — ~3 import paths to update

M10: Absorb routines into config

orchestrator/routines/ → orchestrator/config/routines/
555 LOC routine discovery and YAML loading. Already depends only on config.enums and config.models. Both modules answer “what is the system configured to do?” — config defines the schema, routines loads and discovers instances of that schema. Consumed by api, cli, and app.py. Merging eliminates the quasi-circular risk where config imports from routines for type references.
Medium effort — ~14 import paths to update, but mechanical find-replace

Phase 3: Internal cleanup — reduce fan-out within surviving modules

M11: Extract ReviewService from API router

Business logic in api/routers/review.py → git/review_service.py
The review router directly imports from 5 git sub-modules, workflow.event_logger, and workflow.service. After M3+M5, all review-related code lives in git. Extracting a ReviewService (diff orchestration, prune coordination, test dispatch) reduces the router to thin HTTP handling with 1–2 imports.
Medium effort — ~400 LOC to extract

M12: Extract RunService from API router

Business logic in api/routers/runs.py → workflow/run_service.py
runs.py is the widest router — it directly imports from 15+ modules to orchestrate run creation, lifecycle transitions, activity queries, and branch management. A RunService would encapsulate these operations and reduce the router to request/response mapping. This is the highest-impact single change for API boundary health.
High effort — largest refactor, highest payoff for API boundary cleanup

Key Architectural Patterns Design decisions and abstractions

The core patterns that define the system's architecture.

Request FlowHow an API request flows through the system

Registry-Based Agent Architecture

All agent types implement the AgentRunner protocol and self-register via agent_factory.register(). Each agent lives in its own sub-package under runners/agents/ with agent, factory, config, and parser modules. Auto-discovery via pkgutil.iter_modules loads all agents at startup — adding a new agent requires only creating a new sub-package.

Event-Driven State Machine

WorkflowEngine emits immutable WorkflowEvents on every state change. Events are persisted to the EventStore and broadcast via WebSocket. This enables audit trails and real-time UI updates.

Callback-Based Agent Output

Agents report progress via typed callbacks (on_checklist_update, on_submit, on_output). The executor translates callbacks into state mutations, decoupling agent logic from orchestration.

Separated State & Persistence

In-memory state (state.models.Run) is distinct from DB models (db.models.RunModel). SessionStateManager manages mutable state; repositories handle persistence. This keeps the engine pure.

Dependency Injection

api/deps.py provides factory functions for every service. Each request gets fresh instances. No global mutable state. Clean, testable, and thread-safe.

Git Worktree Isolation

Each run gets its own git worktree, providing filesystem isolation between concurrent agent executions. WorktreeManager handles creation, cleanup, and recovery after server restarts.
