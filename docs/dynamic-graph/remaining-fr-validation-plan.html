Dynamic Graph Remaining FR Validation Plan

Status update for the typed work graph effort. The ledger in docs/dynamic-graph/status.md remains the source of truth. This page explains what is still unproven, why progress has been slow, and the smallest scenarios that should close the remaining functional requirements.

Where We Are

The graph implementation has substantial product evidence: real /api/runs dogfood runs have created planner-authored topology, moved work through worker, verifier, gap planner, corrective worker, and deterministic check nodes, captured file state, enforced callback validation, recovered a dead check lease, completed accepted regions, and rejected unsupported graph runners before seeding graph events.

The same dogfood runs also found bugs that deterministic tests missed: missing verifier requirement packets, stale final-check acceptance, validator citation mismatch, ignored tool-cache staging, stale recovery duplicate-position races, API state incoherence, scheduler waiting_resources readback gaps, and incomplete node-detail control summaries.

Why Progress Has Been Slow

  • The implementation is validating an event-sourced runtime, not a pure library. Many failures only appear when API state, graph projections, runner callbacks, worktree staging, database writes, and scheduler timing interact in the same product path.
  • Tests were too good at proving isolated functions and too weak at proving the product contract. They did not consistently compare public /api/runs state with /graph projections, capture transient scheduler states, or replay restart and stale lease races.
  • Dogfooding used real Codex runner work. That finds high-value bugs, but every run consumes time and tokens, and some evidence was lost because transient states such as resource deferral were not captured at the exact moment they occurred.
  • The ledger was previously too willing to treat passing regression tests as validation. The corrected proof standard requires product-real usage plus supporting tests, which exposes more remaining work but prevents false completion.

Systemic Testing Gaps

Gap Effect Repair direction
Function tests without product-path oracles Pure command/projection tests passed while runner packets and API readbacks were wrong. Add product-state fixtures that assert public API, graph projection, and event facts together.
No durable transient-state capture Resource conflict behavior was visible in events but missed by scheduler readback at the live moment. Add pause/checkpoint support or stop-after-position probes for scheduler states.
Restart races treated as unit recovery only Duplicate event-position and SQLite-lock bugs appeared only during server startup recovery. Use copied DB state packs with active leases and restart the real server path.
Incomplete callback matrix Submit, grade, and patch were exercised, but heartbeat, cancellation, artifact/output, and failure callbacks were not. Create tiny callback-specific dogfood runs with minimal topology and no broad feature work.
Stale fixture shape Old summary rows lacked newer control fields, so old DB readbacks could not prove new node-detail behavior. Use old state for replay/readback regressions, but create fresh narrow runs for newly added readback fields.

Minimal Scenarios

1. Completed Smoke State Pack

Copy the completed 5268031d... DB/event state. Use it for cheap API/projection regression of terminal readbacks, accepted region, final blockers, file-state citation, and no tool-cache staging.

Covers FR-01, FR-03, FR-04, FR-06, FR-14, FR-15, FR-17, FR-18 regression proof.

2. Live Resource Conflict Checkpoint

Reuse the two-writer routine shape from 3c5bfc81..., but stop after the first write lease and second deferral. Query /graph/scheduler, node detail, and bounded events before allowing the first worker to release.

Covers FR-10, FR-15, FR-17 product proof for waiting_resources.

3. Stale Recovery State Pack

Snapshot a DB with an active stale lease and recovery report pending. Start the real server path and prove recovery skips already-revoked leases without duplicate graph positions or database locks.

Covers FR-12 product proof for restart/re-entry durability.

4. Decision and Authority Gate

Seed a narrow graph with a human_gate and authority_request. Query pending decision readbacks, grant/deny authority, and prove downstream readiness changes only after the accepted decision.

Covers FR-02, FR-05, FR-08, FR-10, FR-16, FR-17.

5. Invalid Patch Probe Matrix

Through the runner callback path, submit one invalid patch at a time: unauthorized actor, stale base position, hidden command, unsafe resource mutation, and active-node retire. Assert rejection diagnostics in /graph/patches.

Covers FR-07, FR-08 product rejection proof.

6. Failed Final Invariant

Use a tiny run where worker/verifier evidence exists but the final deterministic check fails. Prove the run stays non-completed with explicit final blockers, then repair the evidence and prove completion.

Covers FR-01, FR-13, FR-14, FR-18.

7. Callback Lifecycle Matrix

Create callback-specific runs for heartbeat, artifact/output, failure, and cancellation. Keep topology small enough that the scenario tests the callback contract rather than a full dynamic feature.

Covers FR-12, FR-16.

8. Edge Binding Topology

Use a deterministic topology with fan-out, join, optional edge, bind_all, and supersede. Assert topology readback contains policies, bound record IDs, and positions.

Covers FR-06, FR-11, FR-17.

9. File Cleanup and Scoped Writes

Run one path-scoped non-conflicting write scenario and one revoked write lease scenario. Prove cleanup is visible as graph work and file-state evidence.

Covers FR-15 and related recovery safety.

How To Reduce Busy Work

  • Convert expensive dogfood DBs into named state packs for readback, projection, and restart regression. Use real product runs only when the FR requires proving a live callback, runner, scheduler, or worktree path.
  • Add precise checkpoints such as "stop after graph position N" or "pause after first lease" so transient proof can be captured without rerunning a full feature.
  • Keep each product scenario tied to one or two ledger rows. Do not use a large dynamic feature run to prove a narrow API readback or rejection behavior.
  • Label evidence honestly: copied state and seeded fixtures are regression proof unless the scenario exercises the same product path a user or downstream runner would use.
