What happened

The parent finished useful setup, launched a child smoke run, observed valid child evidence, then attempted to launch the next child while the prior child was still unresolved. The platform rejected that transition and the agent paused for a human decision.

Why the spend spiked

Every later parent pass carried a large oversight snapshot, long slice briefs, and prior evidence through a high-end model. Mechanical checklist, grading, and retry work then repeated over that enlarged context.

Main finding

The waste was structural: the routine assumed parallel children, the platform enforced sequential children, and each phase carried too much parent state.

Root cause

The parent routine and platform state machine disagreed about child concurrency

The pause point was a useful defect signal: routine intent said parallel; platform behavior said sequential.

Routine plan

Select INV-R-002 while INV-R-001 is still in flight, because the brief explicitly described parallel execution.

mismatch

Actual platform rule

Reject create_child_run when another linked child remains unresolved.

pause

Observed result

Failed create attempt, then clarification request asking how to resolve the conflict.

The fix is not to train the agent to be more patient. The platform needs to expose the concurrency invariant up front, and the routine needs to encode it before slice selection.

Evidence ledger

The run was wasteful, but it still proved several orchestration surfaces

Separate the good evidence from the process waste so the next run can keep the signal and remove the drag.

Keep

  • Parent created a child from the investigation_only template.
  • Child start was enqueued automatically.
  • Child wrote schema-valid run.evidence.v1.
  • Parent fetched and refreshed evidence from child worktree.
  • Pause surfaced a real state-machine/routine contract gap.

Remove

  • Large Markdown state repeated through every fresh phase.
  • Child evidence readiness confused with terminal acceptance readiness.
  • Second-child selection before resolving first child.
  • High-end model used for deterministic MCP bookkeeping.
  • Tool schema error consumed an extra clarification turn.

Decision principle: Treat orchestration evidence as a compact ledger with explicit state flags; keep rich Markdown as human-readable audit material, not as repeated working prompt.

Controls

Four changes would cut most repeat waste in the next run

The controls are small relative to the amount of recurring token and call spend they prevent.

1

Compact parent state

Persist active child id, evidence-ready flag, terminal flag, merge queue, unresolved inventory, and next allowed action.

2

Sequential child gate

Block SP-02 from selecting a new child while any linked child is unresolved unless the routine declares supported concurrency.

3

Model downshift

Route checklist, grade, submit, and schema-retry turns to a cheaper profile; reserve frontier model for diagnosis.

4

Tool schema preflight

Generate exact MCP argument schemas into the agent prompt or locally validate tool calls before dispatch.

Success metric: a parent pass after child evidence should cost hundreds of tokens, not hundreds of thousands.
