Key observation: two distinct gap categories

Category A — Discovery doc coverage gaps (S4/fan_0, S4/fan_1, S5/fan_1): The discovery doc didn't index step file creation patterns or gate evaluation/RunModel code. These agents would likely be eliminated by expanding the S3-T02 discovery prompt to cover those areas.

Category B — Post-execution context needs (S5/fan_0 ×2, S5/fan_2): S5 cross-check needs to know what S4 actually changed — not just what exists in the codebase. A static discovery doc cannot provide this. These agents are exploring the diff between pre- and post-S4 state and would require a different mechanism (e.g., a git diff summary injected into S5 prompts).

5. Proposed Improvements

Two changes were made simultaneously in V7 (step guidance + discovery doc), confounding attribution. Future experiments should change only one variable at a time. The proposals below are designed as isolated interventions.
P1
Expand discovery doc coverage
What
Augment the S3-T02 discovery prompt to explicitly require indexing: (a) step file creation patterns — how existing step plan files are structured, (b) gate evaluation code and RunModel schema. These are the exact areas where S4/fan_0 and fan_1 still spawned explorers.
Hypothesis
The sub-agent descriptions name specific knowledge gaps. If the discovery doc pre-populates those gaps, agents won't need to spawn explorers to fill them.
Experiment design
Modify S3-T02 prompt only. Run V7+P1. Measure: S4 sub-agent count. Confirmation: 0 S4 sub-agents. Failure: Same sub-agent descriptions still appear (gap was task complexity, not doc coverage).
Target: −2 S4 sub-agents Est. saving: 2.8M SA tokens Quality risk: low
P2
Inject test failure output into retry prompts
What
When a task fails with cli_subprocess exit code 1, include the actual test stdout/stderr from the failed attempt in the retry prompt context. Currently the retry sees only "process exited with code 1".
Hypothesis
fan_2/att2 consumed 34K output tokens (vs 3K for the final passing att3), suggesting the model was generating exploratory repairs rather than targeted fixes. Specific failure output should collapse the search space.
Experiment design
Verify first that failure output isn't already in the retry context (it may be — check the orchestrator's retry logic). If not, add it. Measure: attempts-per-task for tasks that historically retry. Confirmation: avg attempts for retrying tasks drops from ~2.5 to ~1.5. Failure: attempt count unchanged; failure output was already visible.
Target: fewer 3-attempt tasks Est. saving: 1–3M parent tokens Quality risk: medium
P3
Inject S4 git diff summary into S5
What
After S4 completes, generate a structured summary of changed files (git diff --name-only HEAD~N) and inject it into the S5 cross-check prompt. This addresses the Category B gap: S5 needs to know what S4 changed.
Hypothesis
S5's 4 Explore sub-agents (2.76M SA tokens) were exploring post-S4 changes. A diff summary provides this directly without exploration. This is a different mechanism than the discovery doc and would need separate validation.
Experiment design
Add a post-S4 hook that writes a diff summary and includes it in the S5 task prompt. Measure: S5 sub-agent count. Confirmation: 0–1 S5 sub-agents. Failure: sub-agents still spawned — they need more than a file list (e.g., they need to read the actual changed code).
Target: −4 S5 sub-agents Est. saving: 2.76M SA tokens Quality risk: medium — may miss regression context
P4
Isolate V7's two changes to confirm attribution
What
Run two ablation experiments: V7-A (step guidance only, no discovery task) and V7-B (discovery task only, no step guidance). This fills the critical confound gap — V7 made two changes simultaneously.
Hypothesis
Current data cannot determine whether the SA reduction from 17→7 was caused by fewer steps (less fan-out → fewer SA opportunities) or the discovery doc (better context → fewer explorers per step). The answer dictates which direction to push next.
Experiment design
V7-A: run with step guidance, without S3-T02 discovery task. V7-B: run without step guidance, with discovery task. Compare SA counts. Confirmation of step-count dominance: V7-A SA ≈ V7 SA. Confirmation of discovery doc dominance: V7-B SA ≈ V7 SA. Mixed result expected — both likely contribute.
Cost: 2 full runs No direct saving — enables targeting of P1–P3 Quality risk: none
P5
Step complexity gate: flag over-scoped steps before fan-out
What
Between S3 (step planning) and S4 (fan-out), add a lightweight scope-check task. It reviews each planned step's file scope and flags any step touching more than N files (suggesting it should be split). This addresses the observed pattern that fan_2/step-03 was significantly more complex than fan_3/step-04 (52× cost difference).
Hypothesis
Over-scoped steps cause high retry rates, which drive large token waste. A scope check before execution can resize steps, distributing complexity more evenly across fan-out children.
Experiment design
Add scope-check task that reads each planned step file and estimates file-touch count. Flag steps above threshold. Measure: retry rate on S4 tasks, and whether cost distribution across fan-out children becomes more uniform. Confirmation: No 3-attempt tasks in S4; cost variance across children drops. Failure: Retry rate unchanged — the issue is code complexity, not step scope.

Important constraint: The checker must be strongly prompted to keep total step count ≤ 5 when splitting. Otherwise it could reverse V7's step-count gains.
Est. saving: 1.5–3M parent tokens if 3-attempt tasks eliminated Checker overhead: ~300–500K tokens Quality risk: medium — splitting may fragment coherent work

7. Synthesis Conclusions

C1 — Step count is the dominant cost multiplier

The number of S4 fan-out children is the strongest predictor of total run cost. Moving from 13 steps (V5) to 4 steps (V7) required only a prompt guidance change. However, step count is also the most volatile parameter — V5 regressed to 13 steps without guidance, and V4 produced only 3. A single prompt phrase controlled a 3× cost swing. This suggests step-count guidance should be treated as a hard constraint, not a soft suggestion.

C2 — Sub-agents are the second-largest cost driver, and a discovery doc reduced them 59%

V5: 17 SA (16.7M tokens). V7: 7 SA (7.6M tokens). All V7 sub-agents were type "Explore" — read-only file exploration. The discovery document prevented exploration in fan_2 and fan_3 but not in fan_0, fan_1, or any S5 task. Caveat: step count also dropped between V5 and V7. Fewer children = fewer SA opportunities, so some of the SA reduction may be mechanical rather than due to the discovery doc quality. P4 (ablation) would resolve this.

C3 — Retry tax concentrates on specific tasks and is asymmetric

In V7, two S4 children (fan_2 and fan_0) and one S5 child (fan_3) required multiple attempts. fan_2 at 3 attempts cost 3.65M parent tokens — 20× more than fan_3 at 67K. All failures were cli_subprocess exit code 1 (test failures), not LLM generation errors. This means the model is correct but the code has bugs. The retry cost asymmetry suggests improving specific agent instructions or providing more targeted failure context would have higher ROI than general prompt improvements.

C4 — V7's SA improvement is real but cannot be cleanly attributed

The 9.1M SA token reduction (V5 → V7) reflects both the step count drop (13→4, fewer tasks to spawn SAs in) and the discovery doc effect. These two changes happened simultaneously. Without the ablation runs (P4), any ROI calculation for the discovery doc is imprecise. The 2.4:1 ROI estimate cited in early analysis drafts is therefore provisional and likely inflated.

C5 — S5 cross-check SA needs are structurally different from S4 SA needs

S4 sub-agents explored static codebase structure (what exists). S5 sub-agents explored dynamic post-execution context (what changed). A static discovery document can address S4's needs but not S5's. S5 requires a mechanism that captures S4's outputs — specifically a git diff or similar artifact. This structural difference means P1 and P3 are addressing different problems and both are needed to reach near-zero SA count.
