What is Task-World?

Task-World is an AI agent orchestration platform that manages the full lifecycle of software engineering tasks — from planning through implementation to verification. It coordinates multiple AI agents (Claude CLI, Claude SDK, Codex, OpenHands) to build code, verify correctness, and iterate on feedback, all within isolated git worktrees.

The system decomposes work into routines (YAML-defined workflows) containing steps (sequential phases) and tasks (parallelizable units). Each task goes through a builder→verifier cycle with auto-verification, grading rubrics, and retry logic. A real-time dashboard provides visibility into runs, diffs, test results, and agent progress.

Multi-Agent Orchestration Registry-based agent architecture with auto-discovery. Supports Claude CLI, Claude SDK, Codex Server, OpenHands (local & Docker), and user-managed agents.
Builder/Verifier Cycle Tasks go through build→verify→grade cycles with configurable max attempts, auto-verification scripts, and LLM-based grading rubrics.
Git Worktree Isolation Each run gets its own git worktree, providing filesystem isolation between concurrent agent executions with branch management.
YAML-Defined Routines Declarative workflow definitions with step conditions, fan-out expansion, repeat-for loops, and embedded routine support.
Real-Time Dashboard React frontend with WebSocket-driven live updates, diff review, test execution, conflict resolution, and activity timelines.
Auto-Verification File existence checks, JSON validation, script execution, and LLM grading with critical/expected/nice priority levels.
Event-Sourced State All state changes emit immutable events persisted to an event store. Enables audit trails, recovery, and real-time UI sync.
MCP Integration Model Context Protocol server exposes tools to external agents for run status, task context, and clarification handling.

Key Architectural Patterns Design decisions and abstractions

The core patterns that define the system's architecture.

Request FlowHow an API request flows through the system

Registry-Based Agent Architecture

All agent types implement the AgentRunner protocol and self-register via agent_factory.register(). Each agent lives in its own sub-package under runners/agents/ with agent, factory, config, and parser modules. Auto-discovery via pkgutil.iter_modules loads all agents at startup — adding a new agent requires only creating a new sub-package.

Event-Driven State Machine

WorkflowEngine emits immutable WorkflowEvents on every state change. Events are persisted to the EventStore and broadcast via WebSocket. This enables audit trails and real-time UI updates.

Callback-Based Agent Output

Agents report progress via typed callbacks (on_checklist_update, on_submit, on_output). The executor translates callbacks into state mutations, decoupling agent logic from orchestration.

Separated State & Persistence

In-memory state (state.models.Run) is distinct from DB models (db.models.RunModel). SessionStateManager manages mutable state; repositories handle persistence. This keeps the engine pure.

Dependency Injection

api/deps.py provides factory functions for every service. Each request gets fresh instances. No global mutable state. Clean, testable, and thread-safe.

Git Worktree Isolation

Each run gets its own git worktree, providing filesystem isolation between concurrent agent executions. WorktreeManager handles creation, cleanup, and recovery after server restarts.

Event & Signaling System Immutable history + one-time control commands

The system uses two distinct mechanisms: Events are immutable records of state changes persisted to SQLite and a JSONL journal, then broadcast to all connected UIs via WebSocket. Signals are one-time control commands (pause, cancel, resume) enqueued to the database and consumed by the executor loop on its next tick.

Event Taxonomy

Workflow Lifecycle

  • RunStatusChanged — DRAFT/ACTIVE/PAUSED/COMPLETED/FAILED; carries pause_reason
  • TaskStatusChanged — PENDING→BUILDING→VERIFYING→terminal; carries start/end commit
  • StepCompleted — all tasks in step reach terminal state
  • StepSkipped — step skipped by condition
  • RunStepBackward — run rewound to earlier step
  • ChecklistGateEvaluated — gate pass/fail with blocking items
  • GradesEvaluated — grade threshold result with per-requirement detail
  • AutoVerifyCompleted — auto-verify script results

Agent Lifecycle

  • AgentChangedEvent — agent runner switched on resume
  • AgentDiedEvent — managed agent process exited unexpectedly
  • AgentOutputEvent — batched stdout lines with line_offset
  • AgentErrorEvent — agent error with type and message
  • TaskReverted — task reverted to phase start on resume
  • HealthCheckEvent — pre-run health check started/completed/failed

Fan-out & Human

  • FanOutSpawned, ChildSpawned, ChildCompleted
  • ChildFailed, FanOutCompleted
  • ClarificationRequested, ClarificationResponded
  • ApprovalRequested, ApprovalDecision

Review Workbench

  • PruneApplied — selective change removal with audit commit
  • TestRunStarted / TestRunCompleted
  • ConflictResolved — conflict file marked resolved
  • BackMergeCompleted / BackMergeReverted
  • AgentFixStarted / AgentFixCompleted

Tech Debt Known legacy patterns, stubs, and mismatched abstractions

These are documented, known issues — not surprises. Each item has a clear resolution path. Severity reflects impact on correctness or maintainability, not urgency.

TD-01 Backward-compat shims after module consolidation — RESOLVED

All shim files removed in phase 3 (commit 38102b4): runners/openhands.py, openhands_docker.py, openhands_common.py, codex_server.py, codex_server_common.py, and orchestrator/executor.py. Parser shims (runners/parsers/claude_parser.py etc.) also removed — parsers live in agents/*/parser.py, lazy-loaded via runners/parsers/__init__.py. Note: runners/executor.py is the real 1600+ LOC executor, not a shim.
✓ Resolved — all callers updated to canonical import paths.

TD-02 Duplicate agent detection systems — RESOLVED

runners/detection/detector.py has been deleted. ToolDetector now lives solely in runners/agent_detector.py. The detection/ sub-package re-exports from agent_detector.py for backward compatibility. The API route (GET /api/agent-runners) imports from agent_detector.py directly.
✓ Resolved — single detection system, duplicate removed.

TD-03 Process-local active-run registry

_active_run_ids in workflow/signals/signals.py is a module-level set[str]. It is lost on server restart and is incompatible with multi-worker deployments. Runs paused mid-transition when the process dies silently lose registry state.
✦ Low impact today (single-process). Becomes a correctness bug if deployment model changes.

TD-04 StepSkipped dual-field inconsistency — RESOLVED

The legacy reason alias has been removed from StepSkipped; only skip_reason remains. Recovery code in db/recovery/recovery.py retains a fallback that maps old reason fields to skip_reason when replaying historical journal events.
✓ Resolved — legacy alias removed, recovery fallback retained for old journal events.

TD-05 scheduled_resume_at is a stub — RESOLVED

The stub _scheduled_resume_check() method has been removed from workflow/signals/runtime.py. The scheduled_resume_at DB column remains for future use but the dead stub code is gone.
✓ Resolved — dead stub removed.

TD-06 InMemoryLockManager never raises LockTimeoutError

LockTimeoutError is defined but InMemoryLockManager lets locks expire passively — it never detects or raises on contention. The pessimistic locking contract is not enforced in practice. Tests document this as a known gap.
✦ Implement contention detection, or document that locking is advisory only.

TD-07 Unimplemented planning phase

PhaseType enum includes plan, gap_check, summarize, human_review in addition to active build/verify. RoutineConfig, StepConfig, and TaskConfig carry a planner_agent field that is never read by the executor.
✦ Implement or prune the unused enum values and fields.

TD-08 Dead Codex HTTP-era code — RESOLVED

create_session_payload() has been removed from runners/agents/codex/common.py. It was dead code from a previous HTTP-based Codex integration.
✓ Resolved — dead function deleted.

TD-09 EventBroadcaster opens a new DB session per output line

EventBroadcaster.emit_log_event() opens a fresh async_sessionmaker session per call to persist AgentOutputEvent batches. Under high-throughput agents this creates many short-lived DB connections. Acknowledged in the module docstring as a known trade-off.
✦ Batch output events or reuse a dedicated session with a short-lived transaction.

TD-10 git/repos.py deprecated parameter — RESOLVED

The deprecated include_remote parameter has been removed from list_branches() in git/repos.py. All callers have been updated to use local_only instead.
✓ Resolved — deprecated parameter removed, all callers updated.

TD-11 RunWorkflow constructor arity — RESOLVED

The 15+ callback parameters have been consolidated into an ExecutorCallbacks dataclass in workflow/signals/runtime.py. AgentRunnerExecutor constructs an ExecutorCallbacks instance and passes it as a single callbacks parameter to RunWorkflow.
✓ Resolved — ExecutorCallbacks dataclass introduced.
