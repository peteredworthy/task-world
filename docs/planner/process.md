# Planning Process: Stage-by-Stage Guide

This document describes each stage of the idea-to-plan workflow in detail.

## Information Priority Hierarchy

When conflicts arise, information is prioritized:

1. **Verified Reality** - Actual system behavior (tests, running code)
2. **Prompt** - Explicit instructions in the current task
3. **Human Feedback** - `[HUMAN]` annotations (never generated by LLM)
4. **Intent** - Feature goals and completion criteria
5. **Design Questions** - Unknowns and decision points
6. **Plan** - Implementation breakdown
7. **Steps** - Detailed task specifications

---

## Stage 1: Initial Plan

**Purpose:** Create foundational planning artifacts that expose gaps early.

### Requirements

- Create `intent.md` - Goal, scope, definition of "complete"
- Create `plan.md` - Milestones, implementation order, references
- Create `design-questions.md` - Open questions with context, options, impact, priority
- Create `architecture.md` - Current architecture, proposed extensions, testing strategy

### Verification

- **Auto-verify:** Check all files exist
- **Verifier:** Evaluate content quality and consistency

### Notes

- Front-load uncertainty - better to discover gaps now than during implementation
- Each artifact should be independently useful
- Cross-reference between artifacts where appropriate

---

## Stage 2: Human Review

**Purpose:** Pause execution for human decision-making.

### How It Works

1. System pauses and presents artifacts for review
2. Human reads artifacts and adds feedback as `[HUMAN]` notes inline
3. Human approves to continue

### Key Rules

- LLM **must not** generate `[HUMAN]` annotations
- Human edits documents directly, adding notes inline
- No file overwrites - only additions

### Example Annotation

```markdown
## API Design

We'll use REST with JSON payloads.

[HUMAN] Consider GraphQL for the query-heavy dashboard views.
```

### Verification

- **Auto-verify:** Check for presence of `[HUMAN]` markers in key artifacts
- Human must explicitly approve to proceed

---

## Stage 3: Plan Refinement

**Purpose:** Integrate human feedback and resolve conflicts.

### Requirements

- Address all `[HUMAN]` feedback notes
- Update plan.md with refinements
- Mark resolved questions in design-questions.md
- Document conflicts in CONFLICTS.md (if any)

### Conflict Handling

If unresolved conflicts exist:
- Document in CONFLICTS.md with details
- Route back to Stage 2 (Human Review) for resolution
- Maximum 3 routing iterations before escalation

### Verification

- All `[HUMAN]` notes addressed
- No unresolved conflicts (or documented and routed back)

---

## Stage 4: Step Planning

**Purpose:** Create detailed step plans with contracts and verification criteria.

### Requirements

For each step in the plan, create `step-XX-plan.md` with:

- **Purpose** - What this step accomplishes
- **Functional Contract** - Inputs, outputs, errors
- **Verification Tests** - How to verify success
- **Dependencies** - What must be complete first
- **Context & References** - Relevant documentation/code

### Approach

- Create all step plans in one pass for consistency
- Consider dependencies between steps
- Each step should be independently verifiable

### Routing

Can route back to Stage 2 if new conflicts are discovered during planning.

---

## Stage 5: Task Breakdown

**Purpose:** Convert step plans into atomic, executable tasks.

### Requirements

For each step plan, create `steps/step-XX.md` containing atomic tasks.

### Task Constraints

- **Atomic:** <5 files changed, <500 lines of code
- **Independent:** Verifiable on its own
- **Front-load tooling:** Research and setup tasks come first
- **Runnable:** System remains runnable after each task
- **Referenced:** Include links to relevant context

### Format

Each task should specify:
- What to do
- Expected outcome
- How to verify
- Context references

---

## Stage 6: Dry Run

**Purpose:** Simulate execution to identify gaps before real implementation.

### Process

1. Walk through each task mentally
2. Note assumptions being made
3. Identify potential gaps or blockers
4. Document findings in `dry-run-notes.md`

### Output: dry-run-notes.md

```markdown
# Dry Run Notes

## Task T-01: Setup Project Structure
- **Simulation:** Would create directories, init files
- **Assumption:** Python 3.11+ available
- **Gap:** None identified

## Task T-02: Implement Data Models
- **Simulation:** Would create SQLAlchemy models
- **Assumption:** Using PostgreSQL
- **Gap:** Need to confirm database choice in architecture.md
```

### Action

If gaps are found, update the step files (YAML is source of truth).

---

## Stage 7: Final Check

**Purpose:** Cross-check all artifacts for consistency.

### Verification Checklist

1. Step files align with plan
2. All dry run gaps addressed
3. Intent fully covered by plan
4. No orphaned decisions or unaddressed unknowns
5. Architecture consistent with implementation

### Output: verification-report.md

Document each check and its outcome.

---

## Stage 8: Final Plan Review

**Purpose:** Human approval before execution.

### Requirements

- Present complete plan summary
- List all step files generated
- Human approves or provides final feedback

### Output

Human approval to proceed to execution.

---

## Stage 9: Execution Ready

**Purpose:** Document that planning is complete.

### Requirements

Create `plan-summary.md` containing:

- How the intent will be satisfied
- Ordered list of step files
- Key decisions made
- Risks and mitigations
- Notes or caveats

### Completion Criteria

- All artifacts consistent
- Human has approved
- Step files ready for execution by coding agents
