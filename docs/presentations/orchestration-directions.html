Architecture Research

Making the Orchestrator
More Adaptable

Three directions for evolving beyond the fixed builder/verifier paradigm — adding flexible phases, smarter failure recovery, and optional workflow stages without drowning in complexity.

3 Options Compared UX Wireframes Effort Estimates Use Cases

What's showing cracks

Structural symmetries that aren't unified

Step-level validation

Checks combined results of all tasks in a step. If any task fails, the step fails. No way for the validator to say "task 2 was the problem, only re-run that."

Task 1 OK Task 2 Fail Task 3 OK → Step Fails

Fan-out child validation

Each child verified independently. Parent gets composite result. Similar pattern, different code paths, different behavior.

Child 1 OK Child 2 Fail Child 3 OK → Parent Fails

The gap

A step-level verifier can't identify which task failed and trigger a targeted retry. It's all-or-nothing at the step boundary, even though the fan-out pattern already supports per-child retry.

Phase rigidity

Current: fixed two-phase pipeline

Build → Verify → Done

Every task follows this path. Recovery exists but as an exception handler, not a first-class phase.

Wanted: flexible phase chains

Plan → Build → Verify → Summarize
Build → Gap Check → Fix Gaps → Verify

Planning before building. Summarization after. Gap analysis that spawns targeted fix agents. Steps with only one phase (script-only, or verify-only for manual work).

Optional/conditional steps

A bug-fix routine triggered regularly should skip the "design" step for simple bugs but include it for complex ones. Currently no way to mark steps as conditional.

Should depth be unbounded?

Arguments for recursive nesting

Temporal's model

Child workflows can nest arbitrarily. A workflow can spawn child workflows that spawn grandchild workflows. Maximum flexibility.

Prefect's subflows

"Infinitely-nested subflow structures for modularity." Any flow can call another flow. Complex conditions become simple if-statements.

The appeal: A routine could decompose a task into sub-tasks, which decompose further. An architect agent plans, a splitter breaks work into pieces, builders execute, a synthesizer combines.

Arguments against

Temporal's own guidance

"Do not use Child Workflows to organize code. Flatten hierarchies where possible."

Costs multiply at each level. Default limit: 2,000 incomplete activities per execution. Debugging nested failures is painful.

UI complexity explodes

Zapier caps conditional nesting at 3 levels. n8n's sub-workflow UI becomes confusing past 2 levels. Each level needs its own progress tracking, error display, and retry controls.

DevEx suffers

Routine authors must reason about arbitrary depth. "Where does my variable resolve?" becomes ambiguous. Testing nested routines requires exponentially more scenarios.

Assessment: Unbounded depth is theoretically elegant but practically harmful. Every successful system either caps depth or discourages nesting. The current 4-level hierarchy (routine → step → task → fan-out child) is a good ceiling. The opportunity is making each level more flexible, not adding more levels.
Option A

Configurable Phase Pipelines

Replace the hardcoded build→verify with a per-task phase sequence

Core idea

Each task defines an ordered list of phases. A phase has a type (build, verify, plan, summarize, gap-check, script, human-review) and can be optional or conditional. The engine walks the phase list instead of hardcoded BUILDING→VERIFYING transitions.

tasks: - id: T-01 title: "Implement auth module" phases: - type: plan profile: ARCHITECT prompt: "Design the auth approach" - type: build prompt: "Implement based on plan" - type: verify rubric: [...] - type: summarize profile: SUMMARIZER prompt: "Write changelog entry" - id: T-02 title: "Fix typo in README" phases: - type: build - type: auto-verify # no LLM verify needed
Key insight: Builder and verifier are just phase types. The current system is a special case where every task has phases: [build, verify]. Making this explicit lets you add planning, summarization, or gap-check phases without new state machine branches.

What changes

  • TaskConfig gains a phases list (defaults to [build, verify] for backward compat)
  • TaskStatus becomes phase-aware: the engine tracks current_phase_index
  • Each phase type has a prompt template and an agent profile
  • Phase output is passed as context to the next phase
  • Transitions are between phases within a task, not just build→verify
  • Verify phase can loop back to any earlier phase (not just build)

Medium — ~2 weeks of focused work

Option B

Gap Analyzer + Targeted Retry

Add a step-level validation agent that can identify failures and spawn fix agents

Core idea

After all tasks in a step complete, an optional step verifier examines the combined output and produces a gap report. Instead of failing the whole step, it can:

  • Retry specific tasks with targeted feedback
  • Spawn fix-up tasks for gaps between tasks (integration issues)
  • Pass if the combined output meets the step's goal
steps: - id: S-01 title: "Implement feature" tasks: [T-01, T-02, T-03] step_verifier: prompt: "Review all task outputs as a whole. Identify integration gaps." actions: - retry_task # re-run a specific task with feedback - spawn_fix # create a new task targeting the gap - pass # combined output is acceptable - fail # unrecoverable max_iterations: 3
Step Verifier Gap Report (LLM output): { "assessment": "T-01 and T-03 work individually but the API contract between them doesn't match", "actions": [ { "type": "spawn_fix", "title": "Align API contract between auth and UI", "context": "auth module exports /api/v2/token but UI calls /api/auth/token. Reconcile the paths.", "depends_on": ["T-01", "T-03"] } ] }

What changes

  • StepConfig gains an optional step_verifier block
  • After all tasks complete, engine runs the step verifier
  • Step verifier can return structured actions (retry, spawn-fix, pass, fail)
  • Spawned fix-up tasks are added to the step and executed
  • Step verifier re-runs after fix-up tasks complete (up to max_iterations)
  • Fan-out parent verification becomes a special case of this pattern

Option C

Optional Steps + Conditional Wiring

Make steps skippable, repeatable, and conditionally triggered

Core idea

Steps gain a condition field that determines if they run. Combined with the existing backward-transition system, this enables adaptive routines that adjust their workflow based on input or intermediate results.

steps: - id: S-01 title: "Triage bug" tasks: [T-classify] # always runs, outputs: severity, complexity - id: S-02 title: "Design solution" condition: when: "{{complexity}} == 'high'" tasks: [T-design] # skipped for simple bugs - id: S-03 title: "Implement fix" tasks: [T-fix] - id: S-04 title: "Write regression test" condition: when: "{{severity}} in ['critical', 'high']" tasks: [T-test] # only for serious bugs - id: S-05 title: "Update changelog" condition: when: "always" repeat_for: "{{bug_ids}}" # runs once per bug in batch mode
Key insight: This doesn't change the engine's core state machine. It adds a thin evaluation layer between steps that checks conditions and decides whether to execute or skip. The existing transition system already supports backward jumps; this adds forward skips.

Condition types

Variable-based

when: "{{complexity}} == 'high'"

Evaluates against run variables (inputs or task outputs). Most common case.

Output-based

when: "steps.S-01.has_failures"

Checks results from previous steps. Enables error-handling steps.

Repeat-for (batch)

repeat_for: "{{bug_ids}}"

Run the step N times, once per item. Like fan-out but at the step level. Each iteration gets {{item}} in scope.

Manual gate

when: "manual"

Pauses and asks the user whether to run this step. Good for expensive or risky steps.

Low — ~1 week, mostly config + step progression logic

The dynamism problem

The tension

Static routines protect quality

Requirements can't be removed mid-run. Steps can't be skipped by a lazy agent. The routine is a contract: the AI must deliver what was specified, verified the way it was specified.

Routine defined → Frozen at start → Executed as-is

This is a feature, not a bug. It prevents the AI from negotiating away its own accountability.

But static routines block useful adaptation

  • A task discovers the work is bigger than expected — can't subdivide
  • Builder realizes a shared utility is needed — can't create a prep task
  • Bug fix reveals a second related bug — can't add it to scope
  • Task needs a specialized sub-agent (e.g., security reviewer) — no way to invoke one through orchestration

The current workaround

Use one routine to build another routine. The builder routine creates a YAML file, then a second run executes it. This works but has a full-restart penalty — no shared worktree, no context carry-over, no single run view.

The tool escape hatch problem

What happens when agents go around the orchestrator

If a builder task can spawn sub-agents via tools (shell commands, MCP calls), those sub-agents:

  • Have no verification — their output isn't graded
  • Have no retry — if they fail, the parent must handle it
  • Have no visibility — no UI tracking, no activity events
  • Have no cost tracking — tokens aren't attributed
  • Have no worktree isolation — they operate on shared state

This is effectively running untracked work inside a tracked system. The orchestration value proposition breaks down.

The core question: How do you let a running task add work to the orchestration graph without letting it remove or weaken existing work?

Design principles

  • Append-only: Add tasks, steps, phases — never remove or modify existing ones
  • Orchestrated: New work goes through the same verification pipeline
  • Visible: Everything shows up in the UI and activity log
  • Bounded: Limits on how much work can be dynamically added
  • Attributable: Clear trail of what requested the expansion and why
Option D

Orchestrated Expansion

Tasks can request new work via the orchestrator, not via tool escape hatches

Core idea

Give running tasks an expansion API — a set of orchestrator-managed actions that let a task request additional work. The orchestrator creates, tracks, and verifies the new work exactly like statically-defined work. The requesting task blocks until expansions complete.

Expansion actions (available to builder agents): add_subtask Create a child task under the current task (like dynamic fan-out, but for a single item) add_peer_task Add a new task to the current step (runs in parallel with other tasks) add_next_step Insert a new step after the current one (for work discovered to be prerequisite) request_agent Spawn a specialist sub-agent as a tracked subtask with its own verification NOT available: remove_task Cannot remove existing tasks remove_step Cannot remove existing steps modify_req Cannot weaken requirements skip_verify Cannot bypass verification
Key constraint: Expansions are append-only mutations to the orchestration graph. The original contract (requirements, verification, steps) is immutable. New work is additive and subject to the same quality controls.

How it works mechanically

1. Agent calls expansion API

Builder agent, during its build phase, calls POST /tasks/{id}/expand with a structured expansion request. This is an MCP tool or REST callback — same mechanism as submit_for_verification.

2. Orchestrator validates & creates

Engine checks budget limits (max expansions per task, per step, per run). Creates the new TaskState/StepState with full config including requirements and verification. Records the expansion provenance (which task requested it, why).

3. Parent task pauses or continues

Two modes: blocking (parent waits for expansion to complete, like fan-out) or fire-and-forget (peer task runs in parallel, parent continues). Blocking expansions use existing FAN_OUT_RUNNING mechanics.

4. Expansion gets full orchestration

The new task goes through build → verify (or whatever phase pipeline is configured). Gets its own attempts, grades, retry budget. Shows up in the UI as a child or peer with an "expanded" badge.

5. Budget enforcement

expansion_limits: max_subtasks_per_task: 5 max_peer_tasks_per_step: 3 max_inserted_steps: 2 max_total_expansions: 10 require_human_approval: false # or true for high-stakes

Effort estimate

Medium — ~2 weeks. Builds heavily on existing fan-out infrastructure.

How D stacks with A, B, C

D A Excellent synergy

Expansion + Phase Pipelines

Dynamically spawned tasks inherit the phase pipeline system. A builder can request a specialist sub-agent that runs plan → build → verify. The expansion API specifies which phases the new task needs. This means request_agent is just add_subtask with a specific agent profile and phase list.

D B Complementary

Expansion + Gap Analyzer

The step verifier's spawn_fix action (Option B) is a specific case of orchestrated expansion (Option D). With D, the fix-up spawning in B gets a proper API and budget system instead of being a special step-verifier-only mechanism. B focuses on when to expand (after composite verification), D provides the how.

D C Independent

Expansion + Conditional Steps

Orthogonal. C controls which static steps run. D adds new steps/tasks at runtime. An add_next_step expansion creates a step that could itself have conditions (though this is an edge case). No conflict.

The anti-pattern it replaces

Agent shells out to sub-agent → Untracked work

Becomes:

Agent calls expand API → Tracked, verified, visible

The agent still decides what to subdivide. The orchestrator ensures the subdivided work meets the same quality bar.

Option D

Safety Model: Append-Only Doesn't Mean Unlimited

What the AI can do

Add subtasks to itself

Builder discovers work needs subdivision. Creates tracked children that go through build/verify. Parent blocks until children complete. Just like fan-out, but triggered at runtime instead of from YAML glob pattern.

POST /tasks/T-01/expand { "type": "add_subtask", "title": "Handle Stripe webhooks", "context": "Implement idempotent webhook...", "requirements": [ {"desc": "Verify signature", "priority": "critical"}, {"desc": "Handle retry dedup", "priority": "expected"} ], "blocking": true }

Add peer tasks to the current step

Builder realizes shared prep work is needed. Adds a task that other tasks in the step can depend on. Non-blocking — runs in parallel.

Insert a step after the current one

Builder discovers prerequisite work. Inserts a new step between current and next. All existing steps remain, the new one is additive. Useful for "I need to refactor X before the integration step can work."

What the AI cannot do

Immutable contract

  • Cannot remove any existing task or step
  • Cannot modify requirements on existing tasks
  • Cannot lower priority of requirements (critical → nice)
  • Cannot skip verification on existing or new tasks
  • Cannot expand beyond the budget limits
  • Cannot expand during verification phase (only during build)

Budget prevents runaway growth

Subtasks per taskDefault: 5
Peer tasks per stepDefault: 3
Inserted steps per runDefault: 2
Total expansions per runDefault: 10

Configurable per routine. High-trust routines get larger budgets. Expansion beyond budget returns 429 (Too Many Requests) with a clear message.

Optional: human approval gate

For high-stakes routines, expansions can require human approval before executing. The expansion request is visible in the UI with the AI's justification. Human approves or rejects.

Agent requests → Human reviews → Approved & tracked

Preventing laziness via expansion

Risk: Could an agent create a subtask that duplicates its own work, effectively delegating instead of doing?

Mitigation: The parent task's requirements still must be met. Expansion adds work, it doesn't transfer the parent's obligations. The parent's verifier still grades the parent's requirements. If the agent creates a subtask but doesn't integrate the result, it fails verification.

Recommendation: keep the ceiling, widen the floor

Don't add depth, add flexibility at each level

Current hierarchy (keep this): Routine └── Step + conditions, repeat_for (Option C) └── Task + phase pipelines (Option A) └── Child (fan-out, unchanged) New cross-cutting capabilities: Step Verifier + gap analysis, spawn fix-ups (Option B) Expansion API + append-only runtime growth (Option D)

Why not unbounded depth?

  • Each level has distinct semantics (ordering, parallelism, verification)
  • Flattening these into a generic "node" loses useful constraints
  • The UI benefits from knowing what level it's displaying
  • Routine authors benefit from a clear mental model
  • Temporal and Prefect both warn against deep nesting in practice

What each level means

RoutineThe goal. A reusable workflow template.
StepAn ordered phase of work. Steps run sequentially.
TaskA unit of agent work. Tasks in a step run in parallel.
ChildA fan-out item. Children are mechanical repetitions.

These distinctions are useful. They constrain the system in ways that make it predictable. A generic recursive node would lose these semantics.

The system has structural symmetries (step vs fan-out validation) implemented as separate code paths, hardcoded phases that limit expressiveness, no way to adapt routines to varied inputs, and a static-only execution model that forces agents to bypass orchestration when they discover emergent work.

Four orthogonal improvements that keep the 4-level hierarchy but make each level more flexible and the whole system more dynamic:

On unbounded depth

Don't do it. Temporal and Prefect both discourage it. The 4-level hierarchy has useful semantic distinctions. Option D gives the feeling of dynamic depth (agents can subdivide work) without the cost (new work is always a leaf at the current level, never a new nesting level).

On dynamism

The key insight is append-only mutation. Let agents add work but never remove or weaken existing obligations. The orchestrator creates, tracks, and verifies all work — dynamic or static — through the same pipeline. This preserves the quality guarantee while enabling runtime adaptation.

Recommended first move: C + D (~3 weeks)

Conditional steps (C) and orchestrated expansion (D) together give the best value/effort ratio. C makes routines adaptive to input. D makes execution adaptive to discovery. Neither touches the core engine transition logic. Both are backward compatible. Both address your specific use case of routines triggered with different bugs where the AI needs to dynamically subdivide work.
