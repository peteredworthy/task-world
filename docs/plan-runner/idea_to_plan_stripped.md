# Stripped-Down Execution Map for LLMs

**First Task:** Copy this execution map into `docs/[feature]/execution-map.md` for reference.

## Global Rules (Always Apply)

- Follow **Core Principles**: Functionality > description, Progressive usability, Contract-first, Structured verification, Runnable system.
- Only header + current task visible.
- Prepend new questions.
- Mark resolutions.
- Respect info priority: Verified reality → Prompt → 🚫 Human feedback (provided externally, not generated by LLM) → Intent → Design-questions → Plan → Steps.
- Keep tasks atomic (<5 files, <500 lines).
- System must stay runnable and pass `uv run pre-commit run --all-files`.

**Note:** Human feedback is not produced by the LLM. The LLM must pause and wait until this input is explicitly given before continuing.

---

## Research & Context (Reusable)

- Outputs: `context/*.md`, `plan-changes.md`, `design-questions.md`, `CONFLICTS.md`.
- Gather: patterns, integrations, errors, testing, DB, config, APIs.
- Use whenever unknowns appear in Stages 1, 3, 4, 5.

---

## Stages

### Stage 1 – Initial Plan

- Inputs: Prompt.
- Outputs: `plan.md`, `intent.md`, `design-questions.md`, `architecture.md`.
- Checklist:

  [ ] Summarize intent.

  [ ] Write stepwise plan.

  [ ] Reference architecture/src/tests/docs.

  [ ] Draft design questions.
- NEXT: Stage 2.

### Stage 2 – Human Review (🚫 Human-only, not executed by LLM)

- Inputs: Stage 1 outputs.

- Outputs: Feedback-marked docs.

- Checklist (for human reviewer):

  - Add `[HUMAN]` notes to&#x20;
    - \`intent.md\`
    - \`plan.md\`
    - \`design-questions.md\`

**LLM Instruction:** Do not perform this stage. Wait for human feedback before proceeding.

- NEXT: Stage 3.

### Stage 3 – Plan Refinement

- Inputs: Reviewed docs.
- Outputs: Updated docs, `CONFLICTS.md` if needed.
- Checklist:

  [ ] Apply feedback.

  [ ] Add new questions.

  [ ] Resolve conflicts.

  [ ] Use research if needed.
- NEXT: Stage 2 (if conflicts) or Stage 4.

### Stage 4 – Step Planning

- Inputs: Plan + research outputs.
- Outputs: `step-xx-plan.md`, `plan-changes.md`.
- Checklist:

  [ ] Define purpose + functionality.

  [ ] Write contract (input/output/errors).

  [ ] Define verification tests.

  [ ] Use context.
- NEXT: Stage 2 (if conflicts) or Stage 5.

### Stage 5 – Task Breakdown

- Inputs: Step plans + research.
- Outputs: `steps-step-xx.md`.
- Checklist:

  [ ] Generate atomic tasks.

  [ ] Front-load tooling/grep tasks.

  [ ] Ensure runnable + tests.

  [ ] Add references into tasks.
- NEXT: Stage 6.

### Stage 6 – Dry Run

- Inputs: Step tasks.
- Outputs: Dry-run notes.
- Checklist:

  [ ] Simulate execution.

  [ ] Document expected outputs.

  [ ] Compare vs intent/plan.

  [ ] Update tasks.
- NEXT: Stage 7.

### Stage 7 – Final Check

- Inputs: All docs.
- Outputs: Verified step files.
- Checklist:

  [ ] Cross-check with intent, plan, design-questions, plan-changes.
- NEXT: Stage 8.

### Stage 8 – Final Plan Review

- Outputs: Full plan summary.
- Checklist:

  [ ] Summarize everything.
- NEXT: Stage 9.

### Stage 9 – Execution

- Inputs: Verified step files.
- Outputs: Completed tasks.
- Checklist:

  [ ] Execute tasks.

  [ ] Ensure linting, typing, tests, formatting pass.
- END.
