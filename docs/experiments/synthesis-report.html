E1 Retry feedback injection with obfuscated failure signal runs unconditionally (targets verified code bug in C6)

Hypothesis (behavioral, matched control)

When a fan-out retry prompt includes previous attempt's error context, the retry agent's first 3 tool calls execute the corrective action. Without feedback injection, the control agent either cannot discover the fix or takes longer to find it.

Deterministic failure design

A custom pre-commit hook fails with only the stderr message "HOOK FAILED — see .hook-diagnostic.log for details". The .hook-diagnostic.log file is in the repo root but git-ignored — the agent cannot discover it via git status. Inside that log: "Create file .sentinel-present in the repo root to pass this hook." The executor's retry path is modified to read this log and inject its contents as child_feedback. The control's retry path does not.

Arms

  • A — Control: No executor modification.
  • B — Experimental: Executor reads .hook-diagnostic.log and injects its contents.
  • C — Ablation: Executor injects a generic hint "check .hook-diagnostic.log if present" regardless of actual content. Isolates "agent followed a hint" from "agent used specific content".

Success criteria

  • B's retry attempt's first 3 tool calls include creating .sentinel-present.
  • A's retry attempt does not. If A's control DOES self-recover (e.g. agent runs ls -a and finds the log), the hypothesis that C6 is the root cause is contradicted.

Architectural avenue — Sandboxed fan-out + merge step

C4 and C5 both trace to a single design decision: fan-out children are responsible for publishing their own output into the repo. Each child writes to docs/{{feature}}/steps/step-XX.md, runs git add && git commit, and hits the pre-commit hook — which triggers pytest, which triggers the mktemp sandbox bug, which kills the agent. Meanwhile, children running in parallel race on the shared worktree's git staging area and end up bundling each other's files into commits with wrong messages.

An alternative pattern that sidesteps both problems:

Correction note. An earlier draft of this section proposed enforcing the sandbox via Claude Code's --allowed-tools / --disallowed-tools flags. That was wrong. Those flags are UI-layer string matchers for the interactive approval prompt, not real containment — any denied tool can be reached via an allowed tool (Bash can write files, so denying Write() while allowing Bash() is pointless). --dangerously-skip-permissions is the correct flag for non-interactive use and must stay. The real write-scope control is an OS-level filesystem sandbox enforced by Claude Code itself, configured via .claude/settings.local.json which the orchestrator already writes into each worktree at git/worktree.py:110–172.
Pattern: narrow the existing sandbox config for fan-out; add a merge step
  1. Each fan-out child writes exactly one artifact to a narrow path inside the worktree, e.g. .orchestrator/fanout/{run_id}/{step}/{item_stem}.md. This is set via the existing output_pattern field in the fan_out config — no schema change. No new gitignore entry needed because any .orchestrator/ artifacts are already untracked (only .orchestrator/state/history.jsonl is gitignored in a targeted way; other files under .orchestrator/ are untracked by default).
  2. Before fan-out starts, narrow the worktree's sandbox by rewriting .claude/settings.local.json with a tighter sandbox.filesystem.allowWrite list. Instead of the current [worktree_abs, "/tmp"], use [{worktree_abs}/.orchestrator/fanout/{run_id}/{step}/, "/tmp", $TMPDIR]. This is the real containment boundary — Claude Code's Bash subprocess cannot write outside these paths, enforced by the OS, not by a string matcher. If a child tries to run git commit, git's write to .git/index fails because .git/ is outside the narrowed allowWrite. No prompt obedience required.
  3. After fan-out completes, restore the worktree's sandbox to the permissive default (whole worktree + /tmp) so that the merge step can write to docs/{feature}/steps/ and run git commit normally.
  4. A sequential merge task runs with normal sandbox. It reads all artifacts from the sandbox subdirectory, copies them to their final repo locations, and performs a single batched git add && git commit. Pre-commit hooks fire exactly once, on the complete batch.
  5. The merge step is the only place that needs hook-awareness. It can use --no-verify for docs-only batches, retry with backoff, or run pre-commit manually with a controlled skip list.

The key insight from your correction: no schema changes, no breaking changes to output_pattern, no changes to factory.py or --dangerously-skip-permissions. The infrastructure already exists — the orchestrator already writes a per-worktree sandbox config at git/worktree.py:_write_sandbox_settings. The only change needed is to make that function callable with a narrower allowWrite list, and to invoke it from the executor before fan-out begins and again after fan-out completes.

What this fixes (if it works)

  • C4 (pre-commit deaths): Eliminated for fan-out children because they cannot reach the pre-commit hook at all — the sandbox blocks writes to .git/index. Only the merge step can encounter a hook failure.
  • C5 (worktree races): Eliminated for writes to docs/. The narrow allowWrite list physically prevents any child from staging or committing anything outside its sandbox subdirectory. Children CAN still step on each other's sandbox files unless we add per-child subdirectories (see open questions).
  • Retry economics: A failing child is now cheap to retry — re-running a single sandbox write doesn't re-do any git work and doesn't fire pre-commit hooks. Only the merge step's retries carry git overhead.

Open design questions

  • Sandbox narrowing granularity: The simplest implementation rewrites the worktree's single .claude/settings.local.json for the duration of the fan-out step so all children share the same narrow allowWrite. This prevents any child from writing to docs/ but leaves children able to overwrite each other's sandbox files. If per-child isolation is needed, either (a) create per-child subdirectories each with their own .claude/settings.local.json (assumes Claude Code walks up from CWD to find the nearest settings file — needs verification), or (b) use per-child worktrees. Option (a) is cleanest if the walk-up behavior exists.
  • Settings restore on fan-out completion: The narrow settings must be restored to the permissive default before the merge step runs, otherwise the merge step can't write to docs/. Save the original JSON before narrowing, restore after. Recovery risk: if the orchestrator process crashes mid-fan-out, the worktree is left with a narrow sandbox. The worktree initialization or auto-resume path should reset the settings file unconditionally on worktree re-entry.
  • Concurrency with non-fan-out tasks: If any non-fan-out task ran in the same worktree concurrently with a fan-out step, the narrowed sandbox would also apply to it. In the current routine, tasks run sequentially within a step so this is not a concern, but the design should document this assumption.
  • TMPDIR fix as a prerequisite (see C4): even with narrowing, the merge step may still hit the /var/folders/ mktemp issue when running pre-commit hooks. Including $TMPDIR in the allowWrite list (both for the narrow fan-out config and the permissive default) would eliminate that root cause. This is a one-line fix that should land before E6.
  • Merge step semantics: Is the merge step a trivial file copy, or does it do sanity-checking (verify each artifact has required sections before copying)? If the latter, it becomes a natural gap-closure point for the C7 problem (dry-run gaps never get applied).
  • How does the child know it's done? Children already rely on the auto_verify test -f {{output_path}} check. With output_path pointing to the sandbox path, this works unchanged.

Architectural avenue — 3-phase implementation / verification / remediation

C6 identified that fan-out retries discard failure context (child_feedback = None on attempt ≥ 2). But the deeper finding from investigating the code is that remediation is not modeled as a distinct phase anywhere in the orchestrator, not just for fan-out. Regular (non-fan-out) tasks that DO receive feedback are run through the same builder prompt template, same executor path, same TaskStatus.BUILDING state — with verifier comments merely prepended as a text section. The agent has to infer that it is remediating from the presence of a "Previous Feedback (Revision Required)" header.

A cleaner workflow model is a 3-phase loop:

Pattern: Implementation → Verification → Remediation mini-loop
  1. Phase 1 — Implementation. Builder agent runs with the full task description and requirements. Its prompt says "build this"; it knows nothing about any prior attempt. This is how the first attempt already works.
  2. Phase 2 — Verification. Verifier agent grades the artifact against the rubric. Outcome is pass or fail. On pass, advance to the next task. On fail, enter Phase 3.
  3. Phase 3 — Remediation. A structurally distinct phase with its own prompt template. The agent is told: "The previous attempt was evaluated and these specific rubric items failed with these specific reasons. Fix those items. Don't re-implement from scratch." The agent receives the list of failing items, grades, and grade_reasons as structured input, not free-form text.
  4. On completion of remediation, return to Phase 2 (re-verify). Loop until pass or max attempts exhausted.

This mirrors the 3-phase shape already documented for the ideal gap analysis pattern (Enumerate → Compare → Close), which is intentional — the same decomposition applies: do the work, check against intent, fix specific gaps. Unifying these under one structural pattern would be a coherent direction across the routine.

Tooling: clone-to-step script

The experiments require a generalized clone script that supports arbitrary start-step indexes and routine variant injection. Because C8 verified that the engine uses run.routine_embedded exclusively at runtime, variants can be applied as dict patches without creating N copies of routine files on disk.

Design: scripts/clone_run_to_step.py

uv run scripts/clone_run_to_step.py \
    --from-run <ref_run_id> \
    --start-step <N> \
    --stop-after-step <M> \            # truncates routine_embedded["steps"] to [:M+1]
    --routine-patch <overlay.yaml> \   # optional; deep-merged into routine_embedded
    --label <experiment_label> \
    --start                              # set run status to ACTIVE immediately

Key implementation points

  • Import WorkflowService before any orchestrator.db imports to avoid the known circular import.
  • Load the routine from disk via discover_routines() by ref_run.routine_id — not from ref_run.routine_embedded, which may be stale.
  • Before saving, apply the YAML patch as a deep merge into new_run.routine_embedded.
  • Pre-mark steps[0..N-1] with synthetic Attempt records (outcome="passed"). Set current_step_index = N.
  • Set source_branch = f"orchestrator/run-{ref_run_id}" so the cloned worktree inherits all committed artifacts from the reference.
  • Termination via routine truncation: If --stop-after-step M is set, slice both new_run.routine_embedded["steps"] and new_run.steps to [:M+1]. Because the engine reads exclusively from routine_embedded (C8), the truncated run has no steps beyond M and check_run_completion() marks it COMPLETED when step M finishes.
  • Termination via gate (alternative): If --pause-after-step M is set instead, append a synthetic step at position M+1 to both lists containing an empty task list and a human_approval gate. The engine pauses at the gate, run stays ACTIVE.

No sidecar watcher is needed. Polling would be wasteful — the engine handles termination natively once the embedded steps list is shaped correctly.
