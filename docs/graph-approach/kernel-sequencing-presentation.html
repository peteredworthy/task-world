Task-World · Execution Graph Kernel

Sequencing the Work

A phased slice plan for building the graph kernel — every slice sized for one of two execution modes, audited by a frontier model, measured from day one, and migrated onto task-world itself as soon as it can carry the load.

Source: execution-graph-prd-plus.md + execution-graph-evaluation.md (§6 amendments) · June 2026

Context

What we're building, and why sequencing is the hard part

The target

The execution-graph kernel from the PRD+, with the agreed amendments: recursive horizon planning, semantic patch revalidation, warn-and-capture file-state with an LLM gatekeeper, git-plumbing snapshots, and carried-over cost/log records.

The constraint

Each step must be executable by one of two modes — a mid-tier model running a builder / verifier / gap-planner sub-agent loop, or the existing idea-to-plan → routine pipeline. Steps too big for either get split until they fit.

The bet we're testing

The agent-with-sub-agents loop is probably more token-efficient, but needs a frontier model "true pass" at the end to catch laziness and lies. Neither mode strongly wins today — so the plan treats mode choice as a measured experiment, not a doctrine.

The endgame

Tuned graph-based task-world should beat both. To get there, the plan dogfoods task-world along the journey and includes explicit analysis steps to find inefficiencies and gaps in its own output.

Execution Modes

Two ways to run a slice — plus the audit that both need

Loop  Sub-agent trio

Mid-tier orchestrator (Sonnet / GPT-5.5 class) drives three sub-agent roles in a cycle until convergence:

  • Builder — implements against the slice spec
  • Verifier — runs fixtures/tests, checks acceptance criteria
  • Gap analyst / planner — diffs delivered vs specified, emits the next work-list

Exit condition: gap analyst reports zero gaps AND verifier reports all criteria green. Cheap iterations, fast feedback, low ceremony.

Routine  Idea-to-plan pipeline

Existing task-world flow: idea-to-plan produces a routine; the routine runs with builder/verifier phases, gates, and auto-verify checks.

  • Front-loaded planning → predictable, traceable
  • Strong boundaries, human gates where wanted
  • Heavier in tokens and ceremony; weak at mid-flight discovery

Best where shallow tests can pass while the semantics are wrong — the planning pass forces the design questions to be answered first.

Audit  Frontier-model true pass — required for both modes

Every slice ends with a frontier-model pass that re-derives the acceptance criteria from the PRD, adversarially reviews the diff, and hunts for laziness and lies (next slide). A slice is done when the audit signs off — not when the loop says it converged or the routine says it completed.

Mode Selection

Rubric: which mode gets which slice

Loop when…

  • The spec is already precise — schemas, transition tables, and fixture formats exist in the PRD (it was written to avoid re-litigating the model)
  • Correctness is mechanically checkable: pure functions, fixtures-first, deterministic outputs
  • Blast radius is bounded to new modules with no cross-cutting wiring
  • The gap analyst can enumerate "done" objectively (e.g. every PRD table row has a fixture)

Routine when…

  • Concurrency or transaction semantics are involved — tests pass easily while the design is subtly wrong
  • The work crosses module boundaries (engine ↔ db ↔ runners ↔ api) and needs an integration map up front
  • Design judgment is part of the deliverable, not just implementation
  • A wrong outcome is expensive to detect late (event-store migration, dispatch atomicity)

Default is Loop — the PRD's precision was bought specifically to make that safe. Routine is the exception, reserved for the four or five slices where being subtly wrong is cheap to ship and expensive to discover.

Tie-breaker: if the gap analyst cannot write an objective "no gaps" definition for the slice, it isn't loop-able — either split it until it is, or send it through the routine.

Quality Control

The audit pass: catching laziness and lies

Mid-tier loops fail in characteristic ways. The audit is a checklist-driven adversarial pass by a frontier model, scoped to one slice, with the PRD and evaluation doc as ground truth.

Laziness (work avoided)

  • Stubbed or pass-through implementations behind green tests
  • Fixtures that assert weaker properties than the PRD table they came from
  • Edge rows silently dropped (e.g. 3 of 9 stale-callback cases covered)
  • Error paths that log instead of reject
  • "Deferred to follow-up" notes that were never in scope to defer

Lies (claims unsupported by the diff)

  • Summary claims a behavior no test exercises
  • "All fixtures pass" where fixtures were edited to pass
  • Determinism claims with hidden clock/random/IO in reducers
  • Renamed-but-unchanged code presented as refactor
  • Acceptance criteria checked off without evidence links

Audit protocol: (1) re-derive the slice's acceptance criteria from the PRD section, ignoring the builder's summary; (2) map each criterion to specific code + test evidence; (3) run the fixture suite fresh; (4) adversarially attempt one violation per invariant (§27.2 list); (5) verdict: accept, accept-with-punchlist, or bounce back to the loop with the gap list. Audit findings are recorded — they are the primary metric for comparing the two modes.

Phase 0

Foundations — durability and instruments first

Nothing here is speculative; all three are prerequisites the evaluation flagged. Crucially, 0.2 is the measurement instrument for the whole mode experiment — it must exist before anything we want to compare.

Why 0.1 is a routine: it migrates the single point of total loss (the evaluation's §4.10) — being subtly wrong is catastrophic and invisible until a crash. Plan-first, gates, human approval on the cutover. 0.2 and 0.3 are bounded, mechanically verifiable — textbook loop slices.

Phase 3

Dynamic planning — the payoff features

SliceScopeModeDone when
3.1 Horizon planner contract Planner node kind; patch = region + successor planner(s) bound to milestone records; planner-generation budget; termination invariant (evaluation §6.1). Design-sensitive: this is the single loop. Routine L Fixture: three-generation planner chain plans → executes → replans on injected verification failure → terminates with no successor.
3.2 Retained planner session Planner session resume with new lease generation; compression between generations; session events (session_compressed etc.) reserved in 0.2/1.1 now wired. Loop M Cost records show resumed planner generations cheaper than fresh-context equivalents.
3.3 Parent/child as planner chain Re-express one real super-parent scenario as planner chain over horizon regions, run through the fixture format and live. Routine M The scenario completes with equivalent outcomes and lower token spend than the parent/child baseline.
3.4 Oversight-layer retirement Migrate remaining parent/child use to planner chains; delete the ~3,400-line oversight/delegation layer. Loop M Layer deleted; test suite green; largest single simplification in the codebase landed.

Gate  Validation gate before 3.4: if 3.3 shows the graph can't cleanly express what parent/child does, stop and redesign — that finding at this point costs one slice; discovering it after deleting the oversight layer costs the layer back.

Phase 2 onward

Dogfooding: task-world builds task-world

The ramp

  • Phases 0–1: slices run as loop / routine directly (task-world graphs don't exist yet). Cost records from 0.2 instrument both modes from the start.
  • Phase 2 (post-2.3): at least one slice per week runs as a graph run in task-world, even if a direct loop would be faster — the point is generating real execution traces.
  • Phase 3+: graph runs are the default carrier. The loop trio maps naturally: builder/verifier are worker/verifier nodes; the gap analyst is a planner node emitting the next horizon. The audit pass becomes an oversight node with a frontier-model profile.

Post-slice analysis (every dogfooded slice)

  • Inefficiency scan: from cost records — token spend per node, context re-injection vs packet assembly, gatekeeper call volume, planner replan count, neutral patch rejections (should be ~zero).
  • Gap scan: audit findings vs slice spec — what the run missed, what boundary checks flagged, where the projection confused a human.
  • Findings file into the backlog as tuning slices — same pipeline, no special process.

Why force it

Tuned graph task-world only wins if it gets tuned, and tuning needs traces of real work — not demos. Building the kernel with the kernel is the cheapest source of honest workload, and every inefficiency found is double value: a fix for the product and a data point for the mode experiment.

Escape hatch

If a dogfooded slice stalls on a task-world defect, fall back to direct loop mode for the slice and file the defect as its own slice. Never block kernel progress on kernel bugs.

The Experiment

Measuring loop vs routine — settling it with data

Protocol

  • Pair comparable slices (similar size/kind — e.g. 1.4 vs 1.5, 2.4 vs 2.6) and run one in each mode where the rubric permits either.
  • Phase-end measurement slice aggregates: total tokens (split by model tier), wall-clock, audit-finding count and severity, bounce-backs, human interventions.
  • Audit findings are the quality currency: a cheap slice with five audit bounces isn't cheap.

Decision points

  • End of Phase 1: enough paired data to set the default mode for Phase 2's loop-eligible slices.
  • End of Phase 3: three-way comparison — loop vs routine vs graph task-world — on dogfooded slices. This is the "has the graph started winning?" check.

Hypotheses on record

TokensQuality risk
Loop + audit Lower — short cycles, mid-tier pricing, frontier only at the end Laziness/lies; caught by audit, costing bounces
Routine Higher — front-loaded planning, fresh context per phase Over-constrained validation; plan staleness
Graph (tuned) Lowest eventually — packet context, retained sessions, deterministic checks Untuned: orchestration overhead per node

Expectation: loop wins early on token cost; the audit-bounce rate decides whether it wins overall. Graph task-world should cross over during Phase 3 once retained planner sessions and packet assembly land — if it doesn't, the measurement slices say why.

Summary

The recommendation in five lines

  1. Default to the sub-agent loop for precise, fixture-checkable slices; reserve the idea-to-plan routine for the five concurrency/integration/design slices (0.1, 1.7, 2.1, 2.3, 3.1).
  2. Every slice ends with a frontier-model audit that re-derives acceptance criteria from the PRD and hunts laziness and lies; audit bounces are the quality metric.
  3. Instruments before experiment: cost records (0.2) land first, so loop vs routine is settled by recorded numbers at phase-end checkpoints, not by intuition.
  4. Fixtures before reducers: the PRD's tables become an audited fixture corpus before implementation starts — the loop's ground truth can't be negotiated down.
  5. Dogfood from the first possible moment (after 2.3/2.6) and analyze every dogfooded run for inefficiencies and gaps — that analysis stream is what turns "graphs should win eventually" into "graphs win, measurably, here."

First three actions

① Run slice 0.2 (cost records) as the first loop experiment — it's small, mechanically checkable, and instruments everything after it.  ② Start 0.1 (event-log durability) through idea-to-plan in parallel.  ③ Draft the audit checklist as a reusable prompt so the true pass is consistent from the first slice.
