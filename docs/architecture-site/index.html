Boundary Concerns Identified coupling and cohesion issues

Areas where module boundaries could be tightened. No critical violations found — the architecture is well-structured with clean layering.

ConcernSeverityDetailsMitigation
Wide API Router Dependencies Medium api/routers/runs.py imports from 15+ modules, creating a wide dependency fan-out at the presentation layer. Already mitigated by api.schemas acting as contracts. Consider a dedicated RunService to reduce direct module access.
WorkflowService Scope Medium Single class handling state machine orchestration, verification, building, and recovery. Estimated 1500+ LOC. Extract sub-services: VerificationService, RecoveryService, CompletionService.
Runners/Executor Coupling Low AgentRunnerExecutor directly calls workflow service methods (submit, complete_verification), binding execution to workflow. Callback-based design already partially abstracts this. Could further decouple with an event bus.
config Circular Risk Low config imports from routines and runners (for enum types), while both import from config. Currently not circular due to import structure. Move shared types (enums) to a dedicated types module to prevent future circularity.
Git Operations Scattered Low Git ops accessed from workflow.service, api.routers.runs, and runners.executor through different entry points. Consider a GitFacade to centralize all git operations behind a single interface.
State → Runners Dependency Low state.models imports ActionLog from runners, creating a dependency from domain models to execution layer. Move ActionLog to state or a shared types module.

Key Architectural Patterns Design decisions and abstractions

The core patterns that define the system's architecture.

Request FlowHow an API request flows through the system

Protocol-Based Agents

All agent types (CLI, Claude SDK, CodexServer, OpenHands, UserManaged) implement a single AgentRunner protocol. The executor dispatches polymorphically — adding a new agent requires only implementing the protocol.

Event-Driven State Machine

WorkflowEngine emits immutable WorkflowEvents on every state change. Events are persisted to the EventStore and broadcast via WebSocket. This enables audit trails and real-time UI updates.

Callback-Based Agent Output

Agents report progress via typed callbacks (on_checklist_update, on_submit, on_output). The executor translates callbacks into state mutations, decoupling agent logic from orchestration.

Separated State & Persistence

In-memory state (state.models.Run) is distinct from DB models (db.models.RunModel). SessionStateManager manages mutable state; repositories handle persistence. This keeps the engine pure.

Dependency Injection

api/deps.py provides factory functions for every service. Each request gets fresh instances. No global mutable state. Clean, testable, and thread-safe.

Git Worktree Isolation

Each run gets its own git worktree, providing filesystem isolation between concurrent agent executions. WorktreeManager handles creation, cleanup, and recovery after server restarts.
