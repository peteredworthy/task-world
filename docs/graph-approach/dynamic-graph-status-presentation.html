Yes — A More Methodical Approach Is Needed

The current loop is O(n) expensive live runs for n contract bugs: each run dies after surfacing one defect. Invert it. Find the bugs in bulk, deterministically, for free; spend real tokens only on the final, intended acceptance run.

Build a deterministic scripted-runner E2E harnessZERO LLM COST

An in-process fake graph runner that plays the full happy path and the known sad paths (rejected-patch retry, gap no-op rejection, executable-retire guard, late-edge backfill, oracle isolation). No model, no quota, no network. Runs in CI in seconds. This is the regression net that should have existed before DG-5.1a — it would have caught a–v as a batch.

Freeze the dynamic-graph contract in one doc + test it

Canonical port names, required edges per region template, per-role tool exposure (planner vs gap_planner), lease-recovery rules, and the completion invariants. Most of the 25 bugs were this contract being discovered run-by-run instead of specified and asserted once. Encode it as table-driven unit tests on the kernel.

Decouple proof from runner quota

Operational correctness is proven by the deterministic harness. Real LLM runners are reserved for the comparison's measurement, not for finding plumbing bugs. Rate-limit pauses (DG-5.1o, 5.2d) should never again block logic progress.

One rule: every live bug becomes a harness case before the rerun

When a live run does surface something the fake harness missed, add the failing case to the harness first, then fix. The next live run can't regress, and latent bugs surface together instead of one-per-run.

Only then spend tokens on DG-5.2GATED

Gate the five-arm comparison behind a green harness. Run each arm once, deliberately, with a real runner, as measurement — not as a debugging tool. Record results or remaining blockers with evidence and call operational status.

Net effect. Steps 1–2 are a few days of deterministic test work that retires most of the live-run spend. The architecture is already proven capable (run 0c053df6 closed the loop). The job now is to make "it works" repeatable and cheap to verify, then run the comparison once.
