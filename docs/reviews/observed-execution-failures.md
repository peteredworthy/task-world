**Started:** 2026-03-05
**Scope:** Failures observed during execution of the `routine-improvements` routine (run a70797d7).

## Systemic Issue: Brittle Pre-Created Auto-Verify Checks

Auto-verify commands are written at plan time by an LLM that generates the routine YAML, then embedded into the run at creation time. The builder agent cannot modify these checks — it can only modify source code in its worktree. When a check is wrong, the builder fails repeatedly until it exhausts retries, then the run pauses.

**This pattern caused 4 of 6 execution failures observed so far (F-01 through F-04).** The checks are never tested against the actual codebase before a run starts. The remaining 2 failures (F-05, F-06) stem from executor lifecycle race conditions — see separate section below.

### Failure Modes

| Failure | Root Cause | Check | Impact |
|---------|-----------|-------|--------|
| Wrong class name | `import Executor` but class is `AgentExecutor` | `executor_imports` (S-03/T-01) | 2 attempts exhausted, run paused |
| Wrong module path | `orchestrator.routers.tasks` but actual path is `orchestrator.api.routers.tasks` | `router_import` (S-10/T-01) | 3 attempts exhausted (recovery agent intervened) |
| Wrong working directory | `cd ui` fails because `worktree_path` pointed to empty dir | `typescript_check` (S-10/T-02) | 2 attempts exhausted, run paused |
| Invalid test fixture | `StepConfig(tasks=[])` but model requires `min_length=1` | `schema_field` (S-11/T-01) | 2 attempts exhausted, run paused |

### Why Builders Can't Self-Correct

1. The auto-verify commands are baked into the embedded routine at run creation — the builder has no mechanism to edit them.
2. The revision feedback tells the builder the check failed, but the builder can only change source code. It cannot make `from orchestrator.routers.tasks import router` succeed when the module is at `orchestrator.api.routers.tasks` — the check itself is wrong.
3. The recovery agent (S-10/T-01 attempt 3) correctly diagnosed the issue and marked the task `skipped`, but this required human intervention to actually fix the routine and reset the task.

### Contributing Factors

- **No pre-flight validation.** Auto-verify commands are not tested before the run starts. A simple dry-run of each check against the codebase at run creation would catch import errors and missing paths.
- **Plan-time LLM guesses at code structure.** The routine YAML is generated by an LLM that doesn't have precise knowledge of module paths, class names, or directory layout. It approximates — `orchestrator.routers.tasks` instead of `orchestrator.api.routers.tasks`.
- **Embedded routine is immutable.** Once a run starts, the routine is frozen. Fixing a bad check requires direct DB surgery and task resets.
- **Worktree path drift.** The `worktree_path` stored in the DB can become stale if the worktree is recreated at a different location (observed: DB said `worktrees/r2`, actual was `/private/tmp/task-world-escalate`).

### Potential Mitigations

1. **Pre-flight check execution.** At run creation, execute every auto-verify command against the worktree. Fail fast with actionable errors before any agent work begins.
2. **Check authoring with codebase access.** Give the plan-generating LLM access to `find`, `grep`, or an LSP so it can verify import paths and class names at YAML generation time.
3. **Mutable auto-verify.** Allow the recovery agent (or human) to patch auto-verify commands on a live run without requiring raw DB edits.
4. **Builder escape hatch.** Let the builder escalate a "bad check" signal when it believes the auto-verify command itself is wrong, rather than burning retries on an unfixable check.

### F-05: Step not marked completed — executor deadlock (S-13 → S-14)

**Date:** 2026-03-05
**Task:** S-14/T-01 stuck in `verifying` for 43+ minutes
**Status before fix:** Run `active` but no agent process running

**What happened:**
S-13 had one task which completed successfully. However, the step itself was never marked `completed=1` and `current_step_index` was not advanced from 12 (S-13) to 13 (S-14). This caused the executor's `_find_next_task` to:

1. Start at `step_index=12` (S-13)
2. Check `step.completed` → false, so it doesn't skip
3. Scan S-13's tasks — all completed, no pending/building/verifying
4. Return `(None, False)` — "no tasks to do"
5. Exit the agent loop

Meanwhile S-14/T-01 was legitimately in `verifying` state (builder finished, auto-verify passed) but the executor never reached it. The verifier agent was spawned initially but died (likely a transient process crash), and no re-spawn was attempted because the executor thought the run was done.

**Root cause:** A race condition or bug in step completion — when S-13's single task completed, the step wasn't marked complete and the index wasn't advanced. This is separate from the auto-verify issue and points to a defect in the step advancement logic.

**Fix:** Manually set `steps.completed=1` for S-13 and advanced `current_step_index` to 13. Pause/resume re-spawned the agent, which immediately completed verification and moved to T-02.

**Date:** 2026-03-05

**Status before fix:** Run `paused` with `pause_reason=agent_health_check_failed`

**What happened:**
The agent process completed T-02's verification and exited normally. The run was paused despite no actual failure — all completed work was fine.

Event sequence at 18:48:37 (all within ~1 second):
1. `grades_evaluated` — T-02 verification passed
2. `task_status_changed` — T-02 → completed
3. `agent_died` — health monitor fires `agent_health_check_failed`
4. `run_status_changed` — active → paused

**Root cause:** The health monitor runs concurrently with the executor loop and doesn't coordinate with task transitions. When the agent process exits after completing a task (normal behavior for CLI subprocess agents), the health monitor interprets it as a failure if the run is still active and tasks remain.

**Fix:** Simple resume — no data correction needed. The executor spawned a new agent for T-03 on resume.

## Systemic Issue: Executor Lifecycle Races

F-05 and F-06 reveal a second class of problems beyond brittle auto-verify checks: **the executor has race conditions in step/task transitions**.

- **F-05 (step not completed):** The step completion logic failed to mark S-13 complete, stranding the run at a stale `current_step_index`. The executor's `_find_next_task` only skips steps where `step.completed=True`, so a step with all tasks completed but `completed=False` creates a deadlock.
- **F-06 (health monitor race):** The health monitor doesn't distinguish between "agent exited because it finished its task" and "agent crashed". When the CLI subprocess agent completes a task and exits, the monitor fires a false alarm if the executor loop hasn't yet advanced to the next task.

### Potential Mitigations

1. **Defensive step completion.** `_find_next_task` should mark a step completed if all its tasks are in terminal states, rather than relying solely on the step completion event having fired.
2. **Health monitor grace period.** After a task completes, suppress health check failures for a short window to allow the executor loop to advance and spawn a new agent.
3. **Health check coordination.** The health monitor should check whether the run has actionable tasks before declaring a failure — if the current task just completed, the agent exit is expected.

## Verifier Overreach: Grading Beyond Stated Requirements

### F-07: Verifier invents requirements when rubric is missing (S-12/T-02)

**Date:** 2026-03-05
**Run:** ff7863ca
**Task:** Implement summary cache and generator
**Status:** `failed` after 2 attempts

**What happened:**
All 3 auto-verify checks passed on both attempts (module imports, wiring grep, 4 summary tests pass). The verifier LLM graded 4 requirements — but penalized for criteria not present in the requirement text.

**Attempt 1 — R4 (critical) graded B, needed A:**
- Requirement: "summarized content appears in prompts for tasks with context_from[summarize=true]"
- Verifier's reason for B: "summarized artifact context would not appear in **verifier prompts**"
- The requirement says nothing about verifier prompts. The verifier invented a scope expansion.

**Attempt 2 — R1 (critical) graded B, needed A:**
- Requirement: "Summary cache stores and retrieves by content hash"
- Verifier's reasons for B: (1) "no dedicated unit tests exist for SummaryCache" (2) "cache is per-request, so it only deduplicates within a single request"
- The requirement says nothing about unit tests or cross-request persistence. The auto-verify already confirmed tests pass. The verifier added its own quality expectations.

The builder fixed R4 between attempts (A on attempt 2) but the verifier regressed R1 from A to B by applying different invented criteria on the second pass.

**Root cause:** Requirements R1–R3 had no rubric entries. The routine's `verifier.rubric` only contained one entry (`R4_wiring`). Without a rubric to constrain grading, the verifier LLM applied its own judgment about what "good enough for an A" means — including test coverage and architectural decisions that aren't in the requirement text.

**Pattern:** When a requirement has a rubric entry, the verifier grades against that rubric. When it doesn't, the verifier grades against its own implicit standards, which are inconsistent across attempts and systematically stricter than the stated requirement.

### Compounding factor: Requirement text didn't capture the actual intention

The verifier was wrong to grade beyond the stated requirements — but the requirements themselves were also too narrow. The `task_context` described "cache keyed by (artifact_path, content_hash), lifetime: run duration" but R1 only said "Summary cache stores and retrieves by content hash." The per-request instantiation (`SummaryCache()` created fresh on every API call in `routers/tasks.py`) means the cache provides no cross-request benefit — it only deduplicates within a single prompt request. This technically satisfies R1 as written, but doesn't deliver the run-duration caching that `task_context` described and the feature was meant to provide.

This is a distinct failure mode from the verifier overreach: **the requirement text failed to encode the actual design intention**. The verifier noticed the gap (per-request scope) but expressed it as a grade penalty on R1 rather than flagging that R1's text was incomplete. The builder implemented R1 as written and passed. Everyone was "correct" against the text they were given, but the feature doesn't work as intended.

**Pattern:** When `task_context` describes behavior that isn't captured in any requirement's `desc` field, the builder will satisfy the requirements without implementing the full intention. The verifier may notice the gap but can only express it as a grade penalty, which looks like overreach since the requirement text doesn't support the penalty.

### Potential Mitigations

1. **Require rubric entries for all critical requirements.** If a requirement is critical (must get A), the routine should include a rubric entry that tells the verifier exactly what to check. Without one, the verifier will invent criteria.
2. **Constrain verifier prompt.** Add explicit instruction: "Grade each requirement ONLY against its description text. Do not penalize for missing tests, architectural preferences, or scope beyond what the requirement states."
3. **Rubric generation at plan time.** When the routine YAML is generated, auto-create a rubric entry for every critical requirement that mirrors its description, preventing the verifier from freelancing.
4. **Requirement-intention alignment check.** At plan time, validate that every specific behavior described in `task_context` is captured by at least one requirement's `desc`. Flag gaps where the context promises something no requirement covers.

### Key Observations

**What the strengthening fixed:**
The routine-plan-strengthening specifically targeted the 3 high-severity dead-code gaps (A6, A11, A13) with wiring requirements, rubric entries, and auto-verify grep checks. Two of three (A11, A13) moved from dead code to functional (at least on the primary path). This validates the hypothesis that better requirements and verification instructions CAN prevent dead-code gaps.

**What the strengthening didn't fix:**
A6 is the outlier. Despite explicit wiring requirements ("compress_clarifications() is called from respond_to_clarification()") and auto-verify grep checks (`grep -q "compress_clarifications" service.py`), the feature still doesn't work. The builder satisfied the letter of the requirements (the function IS called, the grep passes) without satisfying the intent (the output must reach the agent). This is the same requirement-intention gap observed in F-07: the requirement text was too narrow to capture the full wiring chain.

**New weakness exposed:**
A1 regressed — the blocking behavior for `must:true` auto-verify failures was removed. This was not caught because the auto-verify checks only tested that auto-verify runs before the gate, not that it blocks the transition. The verifier may not have noticed because the functional outcome (task returns to BUILDING) is the same. This suggests that **behavioral requirements about state transitions need auto-verify commands that test the transition behavior**, not just the execution order.

**Pattern: "Letter vs Spirit" compliance:**
Both A6 and A1 demonstrate a pattern where the builder satisfies auto-verify grep checks and stated requirements without achieving the intended behavior. Grep checks verify code presence but not code correctness. Requirements that say "X is called from Y" don't ensure the output of X is used. Requirements that say "X runs before Y" don't ensure X blocks Y on failure. Tighter requirements and richer auto-verify (behavioral tests, not just grep) are needed for features that involve multi-step data flow or conditional control flow.
