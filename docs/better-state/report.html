Category 1 — State Machine Transitions
ISSUE-1.1 Duplicate RUN_START signals on concurrent start_run() calls Low Guarded

Two concurrent calls can enqueue two RUN_START signals. However apply_start_run() checks the run is still DRAFT before applying. The second signal finds the run already ACTIVE and is discarded. No duplicate agent spawns possible.

Verdict: Duplicate signal rows are wasteful but harmless. No correctness fix needed.
workflow/service.py
ISSUE-1.2 Resume sets ACTIVE before agent is spawned Low Negligible

The window between ACTIVE being committed and the executor task being created is sub-millisecond. The stale-run sweeper threshold is 120 seconds. The sweeper cannot fire in this window in practice.

Verdict: Theoretical concern only. Sweep interval makes this impossible to trigger accidentally.
workflow/service.py · workflow/signals/consumer.py
ISSUE-1.3 TOCTOU race in fan-out expansion High FIX-7

Two concurrent callers can both read a PENDING parent task before either writes FAN_OUT_RUNNING. No row-level lock guards the read-expand-write sequence. Each caller independently expands the parent, producing duplicate child tasks.

Worst case: Duplicate child tasks; same work executed twice; wasted tokens and conflicting worktree commits.
workflow/service.py
Category 2 — Executor Lifecycle
ISSUE-2.1 Executor can exit without persisting a pause Medium FIX-6

The finally block calls apply_pause_run(). If DB is unavailable at shutdown this call fails and is only logged. The run stays ACTIVE. The stale-run sweeper provides eventual mitigation, but asynchronously.

Worst case: Run stays ACTIVE until sweeper fires; next startup may spawn a new executor against partially-executed state.
workflow/signals/runtime.py
ISSUE-2.2 Heartbeat stale during long agent.execute() calls Already handled

The sweeper's primary liveness check is _running_tasks, not the heartbeat. A run in _running_tasks returns is_running() = True regardless of heartbeat age. The heartbeat is only a fallback for PID-less phases (verification, fan-out). Long agent.execute() calls do not trigger spurious pauses.

ISSUE-2.3 Agent spawn race during startup recovery Low Mitigated

The startup recovery loop runs in a single asyncio coroutine (lifespan context) — it cannot run concurrently with itself. A concurrent API request arriving for the same run_id during the few milliseconds of recovery processing could overwrite the task reference in _running_tasks, leaking the first task. This requires precise timing to trigger.

Verdict: Low real risk. Lifespan serialises recovery; edge case requires exact timing coincidence.

Required fixes

FIX-4Wrap signal handlers in atomic transactions
ISSUE-8.2 · High

Signal handler contract: BEGIN → mutate state → set handled_at → COMMIT; rollback on any exception. handled_at is set if and only if all mutations committed. Redelivery after a crash re-applies against unmodified state. SQLite serialises writes — the transaction contains only DB operations so the lock hold is negligible.

This is the highest-priority fix: the current partial-commit + redelivery combination can produce real duplicate state transitions for non-idempotent handlers.

FIX-6Pre-persist safety pause before executor loop starts
ISSUE-2.1 · Medium

Write PAUSED/reason=executor_not_started to the DB and commit before entering the executor loop. The loop's first action clears the safety pause atomically with its first heartbeat write. If the process is killed before the loop starts, the run is already PAUSED on next startup.

await service.write_safety_pause(run_id, reason="executor_not_started")
await session.commit()
# loop starts — clears the safety pause on first iteration

The existing finally block remains as a belt-and-suspenders fallback.

FIX-7Optimistic version lock via SQLAlchemy version_id_col
ISSUE-1.3 · HighISSUE-9.1 · Low

Use SQLAlchemy's built-in version_id_col. Add a version column to TaskModel and declare __mapper_args__ = {"version_id_col": version}. SQLAlchemy automatically increments on every UPDATE and raises StaleDataError on concurrent modification. Works identically on SQLite and PostgreSQL at the ORM layer — no isolation-level caveats.

Catch StaleDataError: for fan-out expansion discard the duplicate; for clarification/completion surface a 409 Conflict.

FIX-12Record paused_at on in-progress Attempts
ISSUE-4.1 · Medium

Add paused_at (nullable timestamp) and outcome = "paused" to the Attempt model. When pause fires mid-BUILDING or mid-VERIFYING, set these before transitioning the run. On resume:

• continue: re-attach agent to the paused attempt (file state preserved in worktree)
• revert: explicitly mark the attempt outcome = "abandoned" before creating a fresh one

Makes the distinction between paused, abandoned, and failed explicit in the data model.

Hygiene (low priority)

FIX-16Structured GateBlockedEvent with unmet conditions
ISSUE-7.2 · Low

When a gate blocks progress, capture {gate_name, step_id, step_name, unmet_conditions[]} in a gate_blocked_details JSON column and emit a GateBlockedEvent. The ApprovalRequested event already carries step_id — this adds the full condition list to avoid manual DB inspection.

FIX-17Audit log for direct run state mutations
ISSUE-10.2 · Low

Add an audit_log table: (id, run_id, actor, action, details JSON, request_id, ip_address, created_at). Write records in state-mutating API handlers. Use request middleware to inject request_id and ip_address. Complementary to the event log — fills the gap for direct mutations not covered by workflow events.
